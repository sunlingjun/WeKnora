## 二、分层架构详解

### 2.1 接口定义层（Interfaces Layer）

**位置**：`internal/types/interfaces/`

**作用**：定义各层的接口契约，实现依赖倒置原则（DIP），使上层不依赖具体实现。

#### 2.1.1 接口设计原则

1. **接口隔离原则（ISP）**：每个接口职责单一
2. **依赖倒置原则（DIP）**：Service 依赖 Repository 接口，而非具体实现
3. **面向接口编程**：所有依赖都通过接口注入

### 2.2 业务逻辑层（Service Layer）

**位置**：`internal/application/service/`

**作用**：
1. 实现业务规则和逻辑
2. 协调多个 Repository 完成复杂业务
3. 管理事务边界
4. 调用外部服务（文件存储、向量检索等）
5. 异步任务调度（Asynq）

#### 2.2.2 Service 方法实现示例

**创建知识（从文件）**

```go
func (s *knowledgeService) CreateKnowledgeFromFile(
    ctx context.Context,
    kbID string,
    file *multipart.FileHeader,
    metadata map[string]string,
    enableMultimodel *bool,
    customFileName string,
    tagID string,
) (*types.Knowledge, error) {
    // 1. 获取租户 ID（从 context）
    tenantID := ctx.Value(types.TenantIDContextKey).(uint64)
    
    // 2. 验证知识库存在性（调用 KnowledgeBaseService）
    kb, err := s.kbService.GetKnowledgeBaseByID(ctx, kbID)
    if err != nil {
        return nil, err
    }
    
    // 3. 检查文件是否重复（调用 Repository）
    exists, existingKnowledge, err := s.repo.CheckKnowledgeExists(...)
    if err != nil {
        return nil, err
    }
    if exists {
        return nil, &types.DuplicateKnowledgeError{...}
    }
    
    // 4. 上传文件到存储（调用 FileService）
    fileURL, err := s.fileSvc.UploadFile(ctx, file, ...)
    if err != nil {
        return nil, err
    }
    
    // 5. 创建知识记录（调用 Repository）
    knowledge := &types.Knowledge{
        ID: uuid.New().String(),
        TenantID: tenantID,
        KnowledgeBaseID: kbID,
        // ... 设置字段
    }
    if err := s.repo.CreateKnowledge(ctx, knowledge); err != nil {
        return nil, err
    }
    
    // 6. 提交异步任务（文档解析、向量化）
    task, err := s.task.Enqueue(...)
    if err != nil {
        return nil, err
    }
    
    return knowledge, nil
}
```

**根据 ID 查询（带租户隔离）**

```go
func (r *knowledgeRepository) GetKnowledgeByID(
    ctx context.Context,
    tenantID uint64,
    id string,
) (*types.Knowledge, error) {
    var knowledge types.Knowledge
    
    // 必须包含租户 ID 条件（多租户隔离）
    if err := r.db.WithContext(ctx).
        Where("tenant_id = ? AND id = ?", tenantID, id).
        First(&knowledge).Error; err != nil {
        if errors.Is(err, gorm.ErrRecordNotFound) {
            return nil, ErrKnowledgeNotFound  // 自定义错误
        }
        return nil, err
    }
    
    return &knowledge, nil
}
```

**分页查询（支持多条件过滤）**

```go
func (r *knowledgeRepository) ListPagedKnowledgeByKnowledgeBaseID(
    ctx context.Context,
    tenantID uint64,
    kbID string,
    page *types.Pagination,
    tagID string,
    keyword string,
    fileType string,
) ([]*types.Knowledge, int64, error) {
    var knowledges []*types.Knowledge
    var total int64
    
    // 构建查询（租户隔离 + 知识库过滤）
    query := r.db.WithContext(ctx).Model(&types.Knowledge{}).
        Where("tenant_id = ? AND knowledge_base_id = ?", tenantID, kbID)
    
    // 动态添加过滤条件
    if tagID != "" {
        query = query.Where("tag_id = ?", tagID)
    }
    if keyword != "" {
        query = query.Where("file_name LIKE ?", "%"+keyword+"%")
    }
    if fileType != "" {
        if fileType == "manual" {
            query = query.Where("type = ?", "manual")
        } else {
            query = query.Where("file_type = ?", fileType)
        }
    }
    
    // 先查询总数
    if err := query.Count(&total).Error; err != nil {
        return nil, 0, err
    }
    
    // 再查询分页数据
    offset := (page.Page - 1) * page.PageSize
    if err := query.
        Offset(offset).
        Limit(page.PageSize).
        Order("created_at DESC").
        Find(&knowledges).Error; err != nil {
        return nil, 0, err
    }
    
    return knowledges, total, nil
}
```

#### 2.3.3 Repository 层设计要点

1. **租户隔离**：所有查询必须包含 `tenant_id` 条件
2. **错误处理**：将 GORM 错误转换为业务错误（如 `ErrKnowledgeNotFound`）
3. **查询构建**：使用链式调用构建动态查询
4. **分页支持**：统一的分页参数和返回格式

#### 2.4.2 Handler 方法实现示例

**从文件创建知识（HTTP POST）**

```go
// CreateKnowledgeFromFile 处理文件上传创建知识
// @Summary      从文件创建知识
// @Description  上传文件并创建知识条目
// @Tags         知识管理
// @Accept       multipart/form-data
// @Produce      json
// @Param        id                path      string  true   "知识库ID"
// @Param        file              formData  file    true   "上传的文件"
// @Success      200               {object}  map[string]interface{}
// @Security     Bearer
// @Router       /api/v1/knowledge-bases/{id}/knowledges [post]
func (h *KnowledgeHandler) CreateKnowledgeFromFile(c *gin.Context) {
    ctx := c.Request.Context()
    
    // 1. 验证知识库访问权限
    kb, kbID, err := h.validateKnowledgeBaseAccess(c)
    if err != nil {
        c.JSON(http.StatusBadRequest, gin.H{"error": err.Error()})
        return
    }
    
    // 2. 解析文件参数
    file, err := c.FormFile("file")
    if err != nil {
        c.JSON(http.StatusBadRequest, gin.H{"error": "file is required"})
        return
    }
    
    // 3. 解析可选参数
    metadataStr := c.PostForm("metadata")
    var metadata map[string]string
    if metadataStr != "" {
        json.Unmarshal([]byte(metadataStr), &metadata)
    }
    
    // 4. 调用 Service 层
    knowledge, err := h.kgService.CreateKnowledgeFromFile(
        ctx,
        kbID,
        file,
        metadata,
        // ... 其他参数
    )
    
    // 5. 处理错误
    if err != nil {
        // 处理重复文件错误
        if h.handleDuplicateKnowledgeError(c, err, knowledge, "file") {
            return
        }
        c.JSON(http.StatusInternalServerError, gin.H{"error": err.Error()})
        return
    }
    
    // 6. 返回成功响应
    c.JSON(http.StatusOK, gin.H{
        "success": true,
        "data":    knowledge,
    })
}
```

#### 2.4.3 Handler 层职责划分

| **KnowledgeBaseHandler** | 知识库 CRUD、检索 | `KnowledgeBaseService` |
| **SessionHandler** | 会话管理、消息发送 | `SessionService` |

## 三、依赖注入与容器管理

**位置**：`internal/container/container.go`

**作用**：使用 Uber Dig 实现依赖注入，统一管理所有组件的生命周期。

### 3.1 容器构建流程

```go
func BuildContainer(container *dig.Container) *dig.Container {
    // 1. 注册基础设施
    must(container.Provide(config.LoadConfig))        // 配置
    must(container.Provide(initDatabase))             // 数据库
    must(container.Provide(initRedisClient))          // Redis
    must(container.Provide(initFileService))           // 文件服务
    
    // 2. 注册 Repository 层
    must(container.Provide(repository.NewKnowledgeRepository))
    must(container.Provide(repository.NewKnowledgeBaseRepository))
    must(container.Provide(repository.NewChunkRepository))
    // ... 更多 Repository
    
    // 3. 注册 Service 层（依赖 Repository）
    must(container.Provide(service.NewKnowledgeService))
    must(container.Provide(service.NewKnowledgeBaseService))
    must(container.Provide(service.NewChunkService))
    // ... 更多 Service
    
    // 4. 注册 Handler 层（依赖 Service）
    must(container.Provide(handler.NewKnowledgeHandler))
    must(container.Provide(handler.NewKnowledgeBaseHandler))
    // ... 更多 Handler
    
    // 5. 注册路由（依赖 Handler）
    must(container.Provide(router.NewRouter))
    
    return container
}
```

**Service 构造函数签名**

### 5.1 文件上传创建知识流程

```

   └── 提交异步任务（调用 Asynq.Enqueue）

   ├── 调用 DocReaderClient.ParseDocument（文档解析）
   ├── 调用 ChunkService.CreateChunks（创建 Chunks）

   └── 调用 RetrieveEngine.Index（索引到向量数据库）

```

### 5.2 会话消息发送流程


## 六、设计模式应用

### 6.1 依赖注入模式（Dependency Injection）

**应用场景**：所有 Service、Repository、Handler 的创建

**实现方式**：Uber Dig 容器

**优点**：
- 解耦组件依赖
- 便于单元测试（可注入 Mock）
- 统一管理生命周期

### 6.2 仓储模式（Repository Pattern）

**应用场景**：数据访问层

**实现方式**：
- 定义 `Repository` 接口（在 `interfaces` 包）
- 实现具体 Repository（在 `repository` 包）

**优点**：
- 隔离业务逻辑和数据库细节
- 便于切换数据源（如从 PostgreSQL 切换到 MySQL）
- 统一数据访问接口

### 6.3 服务层模式（Service Layer Pattern）

**应用场景**：业务逻辑层

**实现方式**：
- 定义 `Service` 接口
- 实现具体 Service，协调多个 Repository

**优点**：
- 封装业务规则
- 管理事务边界
- 协调多个数据源

### 6.4 策略模式（Strategy Pattern）

**应用场景**：
- 检索引擎选择（`RetrieveEngineRegistry`）
- 文件存储选择（`FileService`：Local/MinIO/COS）
- Embedding 模型选择（`EmbeddingService`）

**实现方式**：
```go
// 检索引擎注册表
type RetrieveEngineRegistry interface {
    Register(name string, engine RetrieveEngine)
    Get(name string) (RetrieveEngine, error)
    Search(ctx context.Context, kbID string, params SearchParams) ([]*SearchResult, error)
}
```

### 6.5 插件模式（Plugin Pattern）

**应用场景**：聊天管道（`ChatPipeline`）

**实现方式**：
```go
// Plugin 接口
type Plugin interface {
    OnEvent(ctx context.Context, eventType EventType, chatManage *ChatManage, next func() *PluginError) *PluginError
    ActivationEvents() []EventType
}

// 注册插件
pipeline.Register(&RewritePlugin{...})
pipeline.Register(&SearchPlugin{...})
pipeline.Register(&RerankPlugin{...})
```

**优点**：
- 可扩展的管道处理
- 插件可独立开发和测试
- 支持动态加载插件

### 6.6 工厂模式（Factory Pattern）

**应用场景**：
- `FileService` 工厂（根据配置创建 Local/MinIO/COS 实现）
- `ContextStorage` 工厂（根据配置创建 Memory/Redis 实现）

**实现方式**：
```go
func NewFileService(config *config.Config) (interfaces.FileService, error) {
    switch config.Storage.Type {
    case "local":
        return file.NewLocalStorage(config.Storage.Local)
    case "minio":
        return file.NewMinIOStorage(config.Storage.MinIO)
    case "cos":
        return file.NewCOSStorage(config.Storage.COS)
    default:
        return nil, errors.New("unsupported storage type")
    }
}
```

## 七、总结

### 7.1 分层架构优势

1. **职责清晰**：每层只负责自己的职责
2. **易于测试**：可通过接口 Mock 进行单元测试
3. **易于扩展**：新增功能只需实现对应接口
4. **易于维护**：修改实现不影响其他层

### 7.2 关键设计原则

1. **依赖倒置**：上层依赖接口，不依赖具体实现
2. **单一职责**：每个类/接口只负责一个功能
3. **开闭原则**：对扩展开放，对修改关闭
4. **接口隔离**：接口设计精简，避免臃肿

### 7.3 学习建议

1. **从接口开始**：先理解 `interfaces` 包中的接口定义
2. **追踪调用链**：从 Handler → Service → Repository 追踪代码
3. **理解依赖注入**：理解 `container.go` 中的依赖注册
4. **实践调试**：在关键方法打断点，观察调用流程
