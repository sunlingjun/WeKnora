# 网页 URL 导入后 `knowledges.title` 为空：原因与补充方案

## 1. 现象

通过「网页类型 / URL 导入」创建的知识（`knowledges.type = 'url'`，非直链文件 `file_url`）在数据库中 **`title` 常为空**，列表与检索展示不友好。

---

## 2. 根因（源码）

### 2.1 后端：仅透传请求里的 `title`

`CreateKnowledgeFromURL` 在**非** `file_url` 分支创建记录时，直接把入参 `title` 写入 `Knowledge.Title`，**没有**其它兜底（`internal/application/service/knowledge.go`）。

对比：**`file_url` 分支**在 `Title` 为空时会用 `displayName`（文件名 / URL 片段）回填，行为不一致。

### 2.2 前端：未传 `title`

`KnowledgeBase.vue` 中 URL 导入确认逻辑调用：

```ts
createKnowledgeFromURL(kbId.value, { url, tag_id: tagIdToUpload })
```

**未携带** `title` 字段；接口虽支持 `json:"title"`，但实际始终为空字符串。

### 2.3 异步解析：未回填标题

`ProcessDocument` 在 URL 模式 `convert` 成功后，**不会**根据页面真实标题更新 `knowledge.Title`。

- `ReadResult` 含 `Metadata map`，但当前 DocReader 网页链路（`WebParser` + Trafilatura）**没有把页面标题单独写入 gRPC 的 `metadata`**；正文里可能带 `with_metadata=True` 的 markdown 前缀，但 Go 侧未解析用于更新 `title`。

---

## 3. 已落地改动（同步兜底）

在 **`CreateKnowledgeFromURL` 的 `type=url` 分支**：若 `title` 去空白后为空，则使用 **`defaultTitleFromWebURL(url)`** 生成默认标题，规则为：

- 能解析 URL 时：`{hostname} / {路径最后一段}`，最后一段超过约 80 个 rune 则截断加 `...`；
- 无路径时：仅用 `hostname`；
- 解析失败：退回整段 URL 字符串。

这样**无需改前端**即可避免 `title` 完全为空；与 `file_url` 的「无标题则用显示名」思路一致。

---

## 4. 可选增强方案（按需迭代）

| 方案 | 说明 | 代价 |
|------|------|------|
| **A. 前端预填 + 必填（已落地）** | `KnowledgeBase.vue`：「显示标题」随合法 URL 按 `hostname / 路径末段` 预填并可改；**必填**；请求带 `title`、`channel: web`。 | 多一个表单项。 |
| **B. DocReader 返回 `metadata["title"]`** | Python 在抓取后用 Trafilatura `extract_metadata` 或解析 `<title>` / `og:title`，写入 `ReadResponse.metadata`；Go 在 `ProcessDocument` 中若 `title` 仍为空或仍为「仅 URL 兜底」时再更新。 | 需改 docreader + proto 已有字段即可传 map；注意 SSRF/长度与编码。 |
| **C. Go 解析 markdown 元数据** | 若正文以 YAML front matter（Trafilatura `with_metadata=True`）开头，在 `convert` 后解析 `title:` 更新 DB。 | 依赖解析格式稳定性，需防误判。 |
| **D. 解析完成后用 LLM 起标题** | 仅当标题仍为空时，用极小模型根据首段生成短标题。 | 成本、延迟与失败兜底需设计。 |

推荐优先级：**后端同步兜底** + **前端 A** 已具备 → 可选 **B（真实页面标题）** → C / D。
