## 一、账号体系打通（CAS 单点登录集成）

### 1.1 需求概述

平台使用自研 CAS 单点登录体系，需要与 WeKnora 系统打通，实现：
1. 前端页面加载时优先请求 CAS 接口验证登录状态
2. 未登录时跳转到 CAS 登录页面
3. 已登录时自动绑定平台租户及用户
4. 支持 APP 服务绑定和认证信息补全
5. 前端登录有效期 2 小时

### 1.2 CAS 接口说明

#### 1.2.0 环境配置

**环境区分**：CAS 系统支持生产环境和测试环境，配置如下：

**退出登录地址格式**：`{CAS_BASE_URL}cas/logout?service={APP_URL}`

**生产环境退出地址示例**：`https://cas.nxin.com/cas/logout?service=https://zsk.nxin.com/`  
**测试环境退出地址示例**：`https://cas.t.nxin.com/cas/logout?service=https://zsk.t.nxin.com/`

**参数说明**：
- `service`: 退出成功后的回调地址（通常是应用首页）

**前端实现方式**：
```typescript
// 使用环境变量配置 CAS 退出地址
const casBaseUrl = process.env.VITE_APP_CAS || 'https://cas.nxin.com/'
const appUrl = process.env.VITE_APP_APP || window.location.origin + '/'
const logoutUrl = `${casBaseUrl}cas/logout?service=${encodeURIComponent(appUrl)}`
window.location.href = logoutUrl
```

**环境变量配置**（`.env` 文件）：
```bash
# 生产环境 (.env.production)
VITE_APP_CAS=https://cas.nxin.com/
VITE_APP_APP=https://zsk.nxin.com/

# 测试环境 (.env.test)
VITE_APP_CAS=https://cas.t.nxin.com/
VITE_APP_APP=https://zsk.t.nxin.com/
```

**说明**：
- `VITE_APP_CAS`: CAS 服务器基础地址（必须以 `/` 结尾）
- `VITE_APP_APP`: 应用地址（退出后的回调地址，必须以 `/` 结尾）
- 退出登录由前端直接跳转到 CAS 退出页面完成，不需要调用后端 API
- CAS 服务器会清除 Cookie 并重定向到 `service` 参数指定的地址

### 1.3 CAS 协议详解

#### 1.3.1 CAS 协议版本

CAS (Central Authentication Service) 支持多个版本，WeKnora 主要使用 CAS 3.0 协议：

| 特性 | CAS 1.0 | CAS 2.0 | CAS 3.0 |
|------|---------|---------|---------|
| 服务票据（ST） | ✅ | ✅ | ✅ |
| 代理票据（PT） | ❌ | ✅ | ✅ |
| 用户属性传递 | ❌ | ✅ | ✅ |
| JSON 响应 | ❌ | ❌ | ✅ |
| REST API | ❌ | ❌ | ✅ |

#### 1.3.2 CAS 认证流程

```
用户浏览器 → WeKnora前端 → CAS服务器 → WeKnora后端
     │            │            │            │
     │ 1.访问资源  │            │            │
     ├───────────>│            │            │

     │<───────────┤            │            │

     ├─────────────────────────>│            │
     │            │            │            │

     │            │            │            │
     │ 5.CAS验证成功，重定向回SP│            │
     │<─────────────────────────┤            │
     │            │            │            │
     │            │ 6.验证服务票据          │
     │            ├─────────────────────────>│
```

#### 1.4.3 CAS 会话管理说明

**说明**：不需要创建 CAS 会话表。原因如下：

1. **CAS 会话信息已存储在 Cookie 中**：
   - `_cas_sid` 和 `_cas_uid` 由 CAS 服务器设置在浏览器 Cookie 中
   - 每次请求浏览器会自动携带这些 Cookie
   - 后端可以直接从 Cookie 读取并调用 CAS API 验证

2. **WeKnora 使用 JWT Token 管理用户会话**：
   - CAS 验证成功后，生成 JWT Token 返回给前端
   - JWT Token 包含用户ID、租户ID等信息，用于后续 API 调用
   - JWT Token 存储在 `auth_tokens` 表中，已经可以管理用户会话状态

3. **验证流程**：
   - 前端每次请求携带 JWT Token（Authorization Header）
   - 后端验证 JWT Token 有效性（从 `auth_tokens` 表查询）
   - 如果 JWT Token 过期，前端重新调用 CAS 验证接口
   - CAS 验证接口会重新验证 Cookie 中的 `_cas_sid` 和 `_cas_uid`

4. **不需要额外存储 CAS 会话的原因**：
   - CAS 会话信息已经在 Cookie 中，无需数据库存储
   - 每次验证都需要调用 CAS API，无法通过数据库缓存
   - JWT Token 已经可以管理用户认证状态
   - 避免数据冗余和维护成本

#### 1.5.3 CAS 认证 Handler

**文件位置**：`internal/handler/cas_auth.go`（新建）

```go
package handler

import (
    "net/http"
    "github.com/gin-gonic/gin"
    "github.com/Tencent/WeKnora/internal/types/interfaces"
    "github.com/Tencent/WeKnora/internal/logger"
    "github.com/Tencent/WeKnora/internal/errors"
)

type CASAuthHandler struct {
    casAuthService interfaces.CASAuthService
    userService    interfaces.UserService
    tenantService  interfaces.TenantService
    config         *config.Config  // CAS 配置
}

func NewCASAuthHandler(
    casAuthService interfaces.CASAuthService,
    userService interfaces.UserService,
    tenantService interfaces.TenantService,
    cfg *config.Config,
) *CASAuthHandler {
    return &CASAuthHandler{
        casAuthService: casAuthService,
        userService:    userService,
        tenantService:  tenantService,
        config:         cfg,
    }
}

// ValidateCASSession 验证 CAS 会话
// @Summary      验证 CAS 会话
// @Description  通过 Cookie 中的 _cas_sid 和 _cas_uid 验证 CAS 登录状态
// @Tags         CAS认证
// @Accept       json
// @Produce      json
// @Success      200      {object}  map[string]interface{}  "验证成功，返回用户和租户信息"
// @Failure      401      {object}  errors.AppError  "未登录或会话过期"
// @Router       /api/v1/cas/validate [get]
func (h *CASAuthHandler) ValidateCASSession(c *gin.Context) {
    ctx := c.Request.Context()
    
    // 1. 从配置获取当前环境的 Cookie 名称
    if h.config == nil || h.config.CAS == nil {
        logger.Errorf(ctx, "CAS config is not initialized")
        c.Error(errors.NewInternalServerError("CAS 配置未初始化"))
        return
    }
    casConfig := h.config.CAS.GetCurrentConfig()
    cookieSID := casConfig.CookieSID
    cookieUID := casConfig.CookieUID
    
    // 2. 从 Cookie 获取 CAS 会话信息（根据环境读取对应的 Cookie）
    casSid, err := c.Cookie(cookieSID)
    if err != nil {
        casSid = ""
    }
    casUid, err := c.Cookie(cookieUID)
    if err != nil {
        casUid = ""
    }
    
    if casSid == "" || casUid == "" {
        logger.Warn(ctx, "Missing CAS session cookies")
        c.JSON(http.StatusUnauthorized, gin.H{
            "code": 10011,
            "exception": "please to login",
            "msg": "未获取到登录信息",
        })
        return
    }
    
    // 2. 验证 CAS 会话（调用 CAS 接口）
    casUserInfo, err := h.casAuthService.ValidateCASSession(ctx, casSid, casUid)
    if err != nil {
        logger.Errorf(ctx, "Failed to validate CAS session: %v", err)
        c.JSON(http.StatusUnauthorized, gin.H{
            "code": 10011,
            "exception": "please to login",
            "msg": "CAS 会话验证失败",
        })
        return
    }
    
    // 3. 自动绑定或创建用户
    user, err := h.casAuthService.AutoBindUser(ctx, casUserInfo)
    if err != nil {
        logger.Errorf(ctx, "Failed to bind user: %v", err)
        c.Error(errors.NewInternalServerError("用户绑定失败"))
        return
    }
    
    // 4. 自动绑定或创建租户
    tenant, err := h.casAuthService.AutoBindTenant(ctx, casUserInfo)
    if err != nil {
        logger.Errorf(ctx, "Failed to bind tenant: %v", err)
        c.Error(errors.NewInternalServerError("租户绑定失败"))
        return
    }
    
    // 5. 生成 JWT Token（用于后续 API 调用）
    // 注意：不需要创建 CAS 会话记录，CAS 会话信息已在 Cookie 中
    // JWT Token 存储在 auth_tokens 表中，用于管理用户认证状态
    token, refreshToken, err := h.userService.GenerateTokens(ctx, user)
    if err != nil {
        logger.Errorf(ctx, "Failed to generate tokens: %v", err)
        c.Error(errors.NewInternalServerError("Token 生成失败"))
        return
    }
    
    // 6. 返回成功响应
    c.JSON(http.StatusOK, gin.H{
        "code": 0,
        "data": gin.H{
            "user":          user.ToUserInfo(),
            "tenant":        tenant,
            "token":         token,
            "refresh_token": refreshToken,
        },
        "msg": "",
    })
}
```

#### 1.5.4 CAS 认证服务实现

**文件位置**：`internal/application/service/cas_auth.go`（新建）

**核心逻辑详细设计**：

1. **ValidateCASSession**：
   - 从配置获取当前环境的 Cookie 名称和接口地址
   - 创建 HTTP 请求，设置 Cookie（模拟浏览器行为）
   - 调用对应环境的 CAS 接口验证会话
   - 处理超时、网络错误等异常情况
   - 如果主接口失败（code != 0），尝试备用接口（2.0版本）

2. **AutoBindUser**：根据 CAS 用户信息查找或创建 WeKnora 用户
   
   **详细实现逻辑**：
   
   ```go
   func (s *casAuthService) AutoBindUser(ctx context.Context, casUserInfo *types.CASUserInfo) (*types.User, error) {
       var user *types.User
       var err error
       
       // 步骤1：用户存在性检查（按优先级查找）
       // 1.1 优先通过 cas_user_id 查找（最可靠，CAS 用户 ID 唯一）
       if casUserInfo.ID != "" {
           user, err = s.userRepo.GetUserByCASUserID(ctx, casUserInfo.ID)
           if err == nil && user != nil {
               logger.Infof(ctx, "Found user by cas_user_id: %s", user.ID)
               goto updateUser
           }
       }
       
       // 1.2 如果未找到，通过 email 查找（CAS 邮箱与 WeKnora 邮箱匹配）
       if casUserInfo.Email != "" {
           user, err = s.userRepo.GetUserByEmail(ctx, casUserInfo.Email)
           if err == nil && user != nil {
               logger.Infof(ctx, "Found user by email: %s", user.ID)
               goto updateUser
           }
       }
       
       // 步骤2：用户不存在，创建新用户
       logger.Infof(ctx, "User not found, creating new user for CAS user: %s", casUserInfo.ID)
       
       // 2.1 生成用户名（检查唯一性）
       username := casUserInfo.LoginName
       if username == "" {
           // 如果 loginName 为空，使用 email 前缀
           username = strings.Split(casUserInfo.Email, "@")[0]
       }
       
       // 2.2 检查用户名是否已存在（处理冲突）
       existingUser, _ := s.userRepo.GetUserByUsername(ctx, username)
       if existingUser != nil {
           // 用户名冲突，添加后缀
           username = fmt.Sprintf("%s_%s", username, casUserInfo.ID[:8])
           logger.Warnf(ctx, "Username conflict, using: %s", username)
       }
       
       // 2.3 检查邮箱是否已存在（理论上不应该发生，因为已经通过 email 查找过）
       if casUserInfo.Email != "" {
           existingUser, _ = s.userRepo.GetUserByEmail(ctx, casUserInfo.Email)
           if existingUser != nil {
               // 邮箱已存在，说明可能是并发请求，返回现有用户
               logger.Warnf(ctx, "Email already exists, returning existing user: %s", existingUser.ID)
               return existingUser, nil
           }
       }
       
       // 2.4 生成密码并哈希（CAS 用户不需要密码，但数据库字段要求非空）
       // 密码默认取手机号后四位，如果手机号为空或格式不对，使用默认密码
       password := "1234" // 默认密码
       if casUserInfo.MobilePhone != "" {
           // 提取手机号后四位（处理脱敏格式，如 "166****8186"）
           mobilePhone := casUserInfo.MobilePhone
           // 移除所有非数字字符
           digits := ""
           for _, r := range mobilePhone {
               if r >= '0' && r <= '9' {
                   digits += string(r)
               }
           }
           // 取最后四位
           if len(digits) >= 4 {
               password = digits[len(digits)-4:]
           }
       }
       hashedPassword, err := bcrypt.GenerateFromPassword([]byte(password), bcrypt.DefaultCost)
       if err != nil {
           return nil, fmt.Errorf("failed to hash password: %w", err)
       }
       
       // 2.5 构建头像 URL（如果需要）
       avatarURL := ""
       if casUserInfo.Avatar != "" {
           // 拼接完整 URL（根据 CAS 配置）
           avatarURL = fmt.Sprintf("https://%s/%s", casImageHost, casUserInfo.Avatar)
       }
       
       // 2.6 创建用户对象
       user = &types.User{
           ID:           uuid.New().String(),
           Username:     username,
           Email:        casUserInfo.Email,
           PasswordHash: string(hashedPassword),
           Avatar:       avatarURL,
           IsActive:     true,
           CASUserID:    casUserInfo.ID,
           CASLoginName: casUserInfo.LoginName,
           CASRealName:  casUserInfo.RealName,
           CASMobilePhone: casUserInfo.MobilePhone,
           CreatedAt:    time.Now(),
           UpdatedAt:    time.Now(),
       }
       
       // 2.7 保存用户到数据库
       if err := s.userRepo.CreateUser(ctx, user); err != nil {
           // 处理唯一约束冲突（并发创建）
           if strings.Contains(err.Error(), "duplicate key") || strings.Contains(err.Error(), "UNIQUE constraint") {
               // 重新查找用户（按优先级：cas_user_id → email）
               if casUserInfo.ID != "" {
                   user, err = s.userRepo.GetUserByCASUserID(ctx, casUserInfo.ID)
               }
               if user == nil && casUserInfo.Email != "" {
                   user, err = s.userRepo.GetUserByEmail(ctx, casUserInfo.Email)
               }
               if user != nil {
                   logger.Warnf(ctx, "Concurrent user creation detected, returning existing user: %s", user.ID)
                   return user, nil
               }
           }
           return nil, fmt.Errorf("failed to create user: %w", err)
       }
       
       logger.Infof(ctx, "Created new user: %s (CAS ID: %s)", user.ID, casUserInfo.ID)
       return user, nil
       
   updateUser:
       // 步骤3：用户已存在，更新 CAS 相关字段
       logger.Infof(ctx, "Updating existing user: %s", user.ID)
       
       // 3.1 更新 CAS 相关字段（如果为空或需要更新）
       needUpdate := false
       if user.CASUserID == "" && casUserInfo.ID != "" {
           user.CASUserID = casUserInfo.ID
           needUpdate = true
       }
       if user.CASLoginName == "" && casUserInfo.LoginName != "" {
           user.CASLoginName = casUserInfo.LoginName
           needUpdate = true
       }
       if user.CASRealName != casUserInfo.RealName {
           user.CASRealName = casUserInfo.RealName
           needUpdate = true
       }
       if user.CASMobilePhone != casUserInfo.MobilePhone {
           user.CASMobilePhone = casUserInfo.MobilePhone
           needUpdate = true
       }
       
       // 3.2 更新头像（如果 CAS 返回了新头像）
       if casUserInfo.Avatar != "" {
           newAvatarURL := fmt.Sprintf("https://%s/%s", casImageHost, casUserInfo.Avatar)
           if user.Avatar != newAvatarURL {
               user.Avatar = newAvatarURL
               needUpdate = true
           }
       }
       
       // 3.3 确保用户是激活状态
       if !user.IsActive {
           user.IsActive = true
           needUpdate = true
       }
       
       // 3.4 如果有更新，保存到数据库
       if needUpdate {
           user.UpdatedAt = time.Now()
           if err := s.userRepo.UpdateUser(ctx, user); err != nil {
               logger.Errorf(ctx, "Failed to update user: %v", err)
               // 不影响主流程，继续返回用户
           } else {
               logger.Infof(ctx, "Updated user CAS fields: %s", user.ID)
           }
       }
       
       return user, nil
   }
   ```
   
   **关键逻辑说明**：
   - ✅ **用户存在性检查**：按优先级查找（cas_user_id → email）
   - ✅ **用户名冲突处理**：如果用户名已存在，添加 CAS 用户 ID 后缀
   - ✅ **邮箱冲突处理**：如果邮箱已存在（并发创建），返回现有用户
   - ✅ **并发创建处理**：处理数据库唯一约束冲突，重新查找用户（按优先级：cas_user_id → email）
   - ✅ **字段更新策略**：只更新空字段或变更字段，避免覆盖用户手动修改的信息
   - 📝 **说明**：登录名（cas_login_name）仅用于存储和显示，不作为用户查找条件

3. **AutoBindTenant**：为 CAS 用户创建或查找 WeKnora 租户
   
   **详细实现逻辑**：
   
   ```go
   func (s *casAuthService) AutoBindTenant(ctx context.Context, casUserInfo *types.CASUserInfo, user *types.User) (*types.Tenant, error) {
       // 步骤1：检查用户是否已有租户
       if user.TenantID > 0 {
           // 用户已有租户，查询并返回

               logger.Errorf(ctx, "Failed to get tenant by ID: %d, error: %v", user.TenantID, err)

           } else if tenant != nil {
               logger.Infof(ctx, "User already has tenant: %d", user.TenantID)
               return tenant, nil
           }
           // 如果租户不存在，继续创建新租户
       }
       
       // 步骤2：用户没有租户或租户不存在，创建新租户
       logger.Infof(ctx, "Creating new tenant for user: %s", user.ID)
       
       // 2.1 生成租户名称
       tenantName := fmt.Sprintf("%s的工作空间", casUserInfo.RealName)
       if tenantName == "的工作空间" {
           // 如果真实姓名为空，使用登录名
           tenantName = fmt.Sprintf("%s的工作空间", casUserInfo.LoginName)
       }
       if tenantName == "的工作空间" {
           // 如果登录名也为空，使用用户名
           tenantName = fmt.Sprintf("%s的工作空间", user.Username)
       }
       
       // 2.2 创建租户对象
       tenant := &types.Tenant{
           Name:        tenantName,
           Description: "默认工作空间",
           Status:      "active",
           Business:    "default",
       }
       
       // 2.3 创建租户（会自动生成 API Key 和设置默认配置）
       createdTenant, err := s.tenantService.CreateTenant(ctx, tenant)
       if err != nil {
           return nil, fmt.Errorf("failed to create tenant: %w", err)
       }
       
       // 步骤3：将用户绑定到租户
       user.TenantID = createdTenant.ID
       user.UpdatedAt = time.Now()
       if err := s.userRepo.UpdateUser(ctx, user); err != nil {
           logger.Errorf(ctx, "Failed to bind user to tenant: %v", err)
           // 不影响主流程，返回租户
       } else {
           logger.Infof(ctx, "Bound user %s to tenant %d", user.ID, createdTenant.ID)
       }
       
       return createdTenant, nil
   }
   ```
   
   **关键逻辑说明**：
   - ✅ **租户存在性检查**：检查用户是否已有租户（`user.TenantID > 0`）
   - ✅ **租户有效性验证**：查询租户是否存在，如果不存在则创建新租户
   - ✅ **租户名称生成**：优先使用真实姓名，其次登录名，最后用户名
   - ✅ **用户绑定**：创建租户后，更新用户的 `tenant_id` 字段
   - ✅ **错误处理**：租户创建失败返回错误，用户绑定失败记录日志但不影响主流程

4. **会话管理**：
   - **CAS 会话**：存储在浏览器 Cookie 中（`_cas_sid`、`_cas_uid`），由 CAS 服务器管理，无需数据库存储
   - **WeKnora 会话**：使用 JWT Token 管理，存储在 `auth_tokens` 表中
   - **验证流程**：
     1. 前端请求携带 JWT Token（Authorization Header）
     2. 后端验证 JWT Token（从 `auth_tokens` 表查询）
     3. 如果 JWT Token 过期，前端重新调用 CAS 验证接口
     4. CAS 验证接口验证 Cookie 中的 `_cas_sid` 和 `_cas_uid`

**CAS 客户端实现示例**：

```go
type casClient struct {
    config *config.CASConfig
    httpClient *http.Client
}

func (c *casClient) ValidateSession(ctx context.Context, casSid, casUid string) (*types.CASUserInfo, error) {
    // 获取当前环境配置
    envConfig := c.config.GetCurrentConfig()
    apiURL := fmt.Sprintf("https://%s%s", envConfig.APIHost, envConfig.UserInfoAPI)
    
    // 创建请求
    req, err := http.NewRequestWithContext(ctx, "GET", apiURL, nil)
    if err != nil {
        return nil, err
    }
    
    // 设置 Cookie（模拟浏览器行为）
    req.AddCookie(&http.Cookie{
        Name:  envConfig.CookieSID,
        Value: casSid,
    })
    req.AddCookie(&http.Cookie{
        Name:  envConfig.CookieUID,
        Value: casUid,
    })
    
    // 发送请求
    resp, err := c.httpClient.Do(req)
    if err != nil {
        return nil, err
    }
    defer resp.Body.Close()
    
    // 解析响应
    var result struct {
        Code int                `json:"code"`
        Data *types.CASUserInfo `json:"data"`
        Msg  string             `json:"msg"`
    }
    
    if err := json.NewDecoder(resp.Body).Decode(&result); err != nil {
        return nil, err
    }
    
    if result.Code != 0 {
        return nil, fmt.Errorf("CAS API error: %s", result.Msg)
    }
    
    return result.Data, nil
}
```

**注意事项**：
- CAS 会话信息存储在 Cookie 中，由 CAS 服务器设置
- Cookie 名称根据环境不同：生产环境 `_cas_sid`/`_cas_uid`，测试环境 `_cas_t_sid`/`_cas_t_uid`
- 接口地址根据环境不同：生产环境 `open.nxin.com`，测试环境 `open.t.nxin.com`
- 后端需要从配置读取 Cookie 名称和接口地址，而不是硬编码

```typescript
      const [key, value] = cookie.trim().split('=')

  const validateSession = async () => {
    loadCASSession()

        authStore.setRefreshToken(response.data.refresh_token)
        isCASLoggedIn.value = true
        return true
      } else {

    const appUrl = process.env.VITE_APP_APP || window.location.origin + '/'

    // 跳转到 CAS 退出页面
    window.location.href = logoutUrl
  }

  return {
    casSid,
    casUid,
    isCASLoggedIn,
    loadCASSession,
    validateSession,
    logout,
  }
```

### 2.3 权限控制方案

#### 2.3.0 权限模型

共享知识库采用基于角色的权限控制（RBAC）模型：

```go
// CheckKnowledgeBasePermission 检查知识库权限
func (s *knowledgeBaseService) CheckKnowledgeBasePermission(
    ctx context.Context,
    kbID string,
    permission string,  // 'read' | 'write' | 'delete'
) (bool, error) {
    // 1. 获取知识库信息
    kb, err := s.repo.GetKnowledgeBaseByID(ctx, kbID)
    if err != nil {
        return false, err
    }
    
    // 2. 获取当前用户 ID
    userID := ctx.Value(types.UserIDContextKey).(string)
    
    // 3. 如果是个人知识库，检查是否为创建者
    if kb.Visibility == KnowledgeBaseVisibilityPrivate {
        return kb.OwnerID == userID, nil
    }
    
    // 4. 如果是共享知识库，检查成员权限
    member, err := s.memberRepo.GetMemberByKBAndUser(ctx, kbID, userID)
    if err != nil {
        return false, err
    }
    
    // 5. 根据权限要求检查角色
    switch permission {
    case "read":
        return member != nil, nil
    case "write":
        return member != nil && (member.Role == KBMemberRoleOwner || member.Role == KBMemberRoleEditor), nil
    case "delete":
        return member != nil && member.Role == KBMemberRoleOwner, nil
    default:
        return false, errors.New("invalid permission")
    }
}
```

## 四、注意事项

### 4.1 CAS 单点登录

1. **安全性**：
   - CAS 会话信息由 CAS 服务器设置在 Cookie 中，浏览器自动携带
   - Cookie 名称根据环境不同：
     - 生产环境：`_cas_sid`、`_cas_uid`
     - 测试环境：`_cas_t_sid`、`_cas_t_uid`
   - 后端从配置读取 Cookie 名称，然后从 `c.Cookie()` 读取会话信息
   - 接口地址根据环境不同：
     - 生产环境：`open.nxin.com`
     - 测试环境：`open.t.nxin.com`
   - JWT Token 过期时间设置为 24 小时（access token）和 7 天（refresh token）
   - CAS 会话信息存储在 Cookie 中，由 CAS 服务器管理过期时间（2 小时）
   - 如果 JWT Token 过期，前端重新调用 CAS 验证接口验证 Cookie 中的会话
   - 注意：Cookie 的 HttpOnly、Secure 标志由 CAS 服务器控制，WeKnora 后端只需读取

2. **环境配置**：
   - 通过配置文件或环境变量 `CAS_ENVIRONMENT` 指定当前环境（production/test）
   - 前端通过 `VITE_CAS_ENV` 环境变量或域名判断（包含 `.t.` 为测试环境）
   - 所有 CAS 相关操作（Cookie 读取、接口调用、登录跳转）都需要根据环境切换
   - 配置示例：
     ```yaml
     cas:
       environment: production  # 或 test
       production:
         api_host: open.nxin.com
         login_host: cas.nxin.com
         cookie_sid: _cas_sid
         cookie_uid: _cas_uid
       test:
         api_host: open.t.nxin.com
         login_host: cas.t.nxin.com
         cookie_sid: _cas_t_sid
         cookie_uid: _cas_t_uid
     ```

3. **兼容性**：
   - 保留现有的用户名密码登录方式（作为备用）
   - CAS 登录和传统登录可以共存

4. **错误处理**：
   - CAS 接口调用失败时的降级策略
   - 网络超时处理
   - 用户信息同步失败处理

### 4.2 共享知识库

1. **性能优化**：
   - 成员数量字段（`member_count`）冗余，避免频繁 COUNT 查询
   - 共享知识库列表查询添加缓存
   - 分页查询优化（使用游标分页）

2. **权限控制**：
   - 所有知识库操作都需要检查权限
   - 共享知识库的知识内容编辑权限也需要检查
   - 前端权限控制不能替代后端权限检查

3. **数据一致性**：
   - 删除知识库时，自动删除所有成员记录
   - 删除用户时，自动删除该用户的所有成员记录
   - 使用数据库事务保证数据一致性
