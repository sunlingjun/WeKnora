# 代码合并执行方案

> **文档创建日期**：2026-04-01  
> **合并目标**：将上游(upstream/main)新增代码合并到本地，保留本地核心功能  
> **核心原则**：合理合并远程新增代码，不改变本地新增逻辑（CAS认证、共享知识库、颜色统一处理、SVG组件）

## 四、分阶段合并执行计划

### 4.1 阶段一：准备与备份（预计30分钟）

```powershell
# 1. 确保工作区干净
git status

# 2. 创建备份分支
git branch backup/2026-04-01-merge-before

# 3. 创建本地功能补丁（以防万一）
git diff upstream/main...HEAD -- internal/handler/cas_auth.go > patches/cas_auth.patch
git diff upstream/main...HEAD -- internal/application/service/cas_*.go > patches/cas_service.patch
git diff upstream/main...HEAD -- frontend/src/assets/theme/theme.css > patches/theme.patch
git diff upstream/main...HEAD -- frontend/src/components/icons/ > patches/svg_icons.patch
git diff upstream/main...HEAD -- internal/application/service/shared_kb.go > patches/shared_kb.patch

# 4. 记录当前提交ID
$currentCommit = git rev-parse HEAD
$upstreamCommit = git rev-parse upstream/main
Write-Host "当前本地提交: $currentCommit"
Write-Host "当前远程提交: $upstreamCommit"
```

### 4.2 阶段二：主干合并（预计2小时）

```powershell
# 1. 开始合并
git merge upstream/main --no-commit --no-ff

# 2. 检查冲突状态
git status

# 3. 如果有冲突，按优先级解决：
#    顺序：theme.css → router.go → main.go → knowledge.go → auth.go → 其他
```

## 五、风险与应对

### 5.1 高风险项

| 风险 | 影响 | 应对措施 |
|------|------|----------|
| CAS认证被覆盖 | 无法登录 | 提前准备补丁文件，冲突时优先保留 |
| 共享知识库逻辑丢失 | 权限混乱 | 重点检查shared_kb.go，确保双模式共存 |
| 主题系统损坏 | UI异常 | 完整备份theme.css，冲突直接采用本地 |
| 数据库迁移冲突 | 数据丢失 | 检查迁移文件顺序，本地迁移号段避开远程 |

### 5.2 回滚方案

```powershell
# 如果合并失败，快速回滚
git merge --abort  # 中止合并
git reset --hard backup/2026-04-01-merge-before  # 回滚到备份

# 或者使用补丁恢复特定文件
git apply patches/cas_auth.patch
git apply patches/theme.patch
```

## 九、总结

本合并方案遵循以下核心原则：

1. **本地核心功能完全保护**：CAS认证、共享知识库、CSS主题、SVG组件
2. **远程新增功能合理合并**：Suggested Questions、Channel、SSRF保护等
3. **冲突解决有优先级**：theme.css > router > main.go > knowledge.go > 其他
4. **充分备份与回滚准备**：分支备份、补丁文件、详细检查清单

建议执行顺序：
1. 先在测试环境执行合并验证
2. 确认无误后再在生产环境执行
3. 合并后进行全面回归测试
