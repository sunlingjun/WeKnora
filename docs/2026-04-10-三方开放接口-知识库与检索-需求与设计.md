# 三方开放接口：知识库与检索（需求与设计）

**版本**：1.5  
**日期**：2026-04-10  
**范围**：WeKnora 三方集成（目标1 + 目标2）

---

## 1. 需求摘要

| 目标 | 能力 | 认证与权限 | 主入口 |
|------|------|------------|--------|
| **目标1** | 用户可见知识库列表；某库下知识列表 | 需登录；沿用现有可见性 | `GET /api/v1/knowledge-bases/user` + `GET /api/v1/knowledge-bases/{kb_id}/knowledge` |
| **目标2** | 按 KB/知识 ID + 问题返回分片，供外部模型 | **无用户登录**；**不做用户归属校验**；按资源 `tenant_id` 检索 | `POST /api/v1/open/knowledge/retrieve`（已实现；`config.open_retrieve` + `X-Open-Retrieve-Api-Key`） |

**目标2 风险**：掌握 **授权 API Key** 与有效 ID 即可取数；Key 须当密码保管，部署侧建议仍配 HTTPS 与基础限流（见 §3）。

---

## 2. 接口约定

### 2.1 目标1

- **`GET /api/v1/knowledge-bases/user`**  
  - Query：`include_shared`（默认 `true`）  
  - 返回：`id/name/type`、角色、统计等（与现网一致）。  
- **`GET /api/v1/knowledge-bases/{kb_id}/knowledge`**  
  - 分页等参数与现网一致；权限同现有 Handler。

### 2.2 目标2（设计稿）

- **`POST /api/v1/open/knowledge/retrieve`**  
  - Header：**`X-Open-Retrieve-Api-Key`**（服务端预先发放的开放检索专用 Key，与用户登录态无关）。  
  - Body：`query`（必填）；`knowledge_base_ids`、`knowledge_ids`（至少其一）；可选 `match_count`。  
  - 响应：`success` + `data[]`（分片字段与 `types.SearchResult` 对齐即可）。  
  - 详细示例见：`docs/api/open-knowledge-retrieve.md`。

## 3. 目标2 安全（简化：授权 API Key）

面向**模型网关 / 后端服务**直连：不做 HMAC、nonce、灰度编排；**强校验仅为「总开关 + 合法 API Key」**，其余限流与审计为可选增强。

| 项 | 要点 | 配置键示例 |
|----|------|-------------|
| **总开关** | 默认关；关闭时直接拒绝，不进入检索 | `open_retrieve.enabled` |
| **授权 API Key** | 请求头 `X-Open-Retrieve-Api-Key` 与配置比对（常量时间比较）；缺失或错误 → `401` | `open_retrieve.api_key` 或 `open_retrieve.api_keys[]` |
| **限流（可选）** | 全局限流或按 Key 限流，防误刷；超限 `429` | `open_retrieve.rate_limit_qps` |
| **审计（可选）** | 结构化日志：`request_id`、状态码、耗时、目标 ID 数量、`result_count`；**不落 `query` 明文**（可记 `query_hash`） | 沿用现有日志 |

**中间件顺序（建议）**：校验 `open_retrieve.enabled` → 校验 `X-Open-Retrieve-Api-Key` →（可选）限流 → Handler。

**错误码（建议）**：`OPEN_RETRIEVE_DISABLED`（403）、`OPEN_RETRIEVE_UNAUTHORIZED`（401）、`OPEN_RETRIEVE_RATE_LIMITED`（429，若启用）、`OPEN_RETRIEVE_INTERNAL_ERROR`（500）。

## 4. 目标2 技术设计（单独 Service）

**组件**：`OpenRetrieveApiKeyMiddleware`（开关 + Key 校验 + 可选限流）→ `OpenRetrieveHandler` → `OpenRetrieveService` + `OpenRetrieveTargetBuilder`。

**核心逻辑**：按主键加载 KB/知识（`GetKnowledgeBaseByID`、`GetKnowledgeByIDOnly`）→ 取资源自带 `tenant_id` 注入上下文 → 组检索目标 → **复用** `HybridSearch`/现有检索管道；**禁止**调用 `validateKnowledgeBaseAccess`、`CheckUserKBPermission`、`GetKnowledgeBatchWithSharedAccess` 及任何依赖登录用户的分支。

**路由**：独立分组、**跳过**全局用户 `Auth`；仅走 `OpenRetrieveApiKeyMiddleware`。

**FAQ**：按 KB `type` 走现有 FAQ 检索路径或与文档分支并行（实现时统一在 Service 内分支）。
