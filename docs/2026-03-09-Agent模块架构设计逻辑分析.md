# Agent 模块架构设计逻辑分析

本文档对 WeKnora 中 Agent 模块的架构、设计逻辑与运行机制进行详细分析，便于理解与扩展。

---

## 1. 概述与设计目标

### 1.1 功能定位

Agent 模块提供**两种运行模式**的统一抽象：

| 模式 | 说明 | 典型场景 |
|------|------|----------|
| **quick-answer** | RAG 模式：检索 → 上下文模板 → 单次 LLM 生成 | 快速问答、知识库问答 |
| **smart-reasoning** | ReAct 模式：多轮 Think → Act（工具调用）→ Observe | 多步推理、工具调用、数据分析 |

同一套「智能体」抽象下通过 `agent_mode` 与配置区分，对话入口统一为 `agent-chat`，由后端根据 `customAgent.Config.AgentMode` 路由到 `KnowledgeQA`（普通模式）或 `AgentQA`（ReAct 模式）。

### 1.2 设计目标

- **双模式统一**：同一套智能体配置与 API，通过 `agent_mode` 区分 RAG 与 ReAct。
- **可配置智能体**：内置智能体 + 自定义智能体（CustomAgent），支持知识库、工具白名单、系统提示词等。
- **工具可扩展**：内置工具 + MCP 工具 + Agent Skills，按需注册、白名单控制。
- **流式与可观测**：思考、工具调用、结果、最终答案均通过 EventBus 推送，由 Handler 转为 SSE 供前端消费。

## 2. 整体架构分层

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                            前端 (Vue 3 + TDesign)                            │
│  AgentSelector / AgentEditorModal / AgentStreamDisplay / AgentShareSettings  │
│  AgentList / AgentSettings / creatChat（选择智能体）                         │
└─────────────────────────────────────┬───────────────────────────────────────┘
                                      │ REST: /api/v1/agents, /api/v1/agent-chat/:session_id
┌─────────────────────────────────────▼───────────────────────────────────────┐
│                           应用层 (Handler + Service)                          │
│  custom_agent handler (CRUD) │ session/qa (AgentQA / KnowledgeQA)             │
│  AgentStreamHandler (订阅 EventBus → StreamManager)                           │
│  AgentService.CreateAgentEngine │ SessionService.AgentQA                      │
└─────────────────────────────────────┬───────────────────────────────────────┘
                                      │
┌─────────────────────────────────────▼───────────────────────────────────────┐
│                         Agent 引擎层 (internal/agent)                         │
│  AgentEngine (ReAct 循环) │ prompts (SystemPrompt) │ EventBus 事件            │
│  tools.ToolRegistry │ tools.* (knowledge_search, web_search, MCP, Skills…)   │
│  skills.Manager (Progressive Disclosure: read_skill / execute_skill_script)   │
└─────────────────────────────────────┬───────────────────────────────────────┘
                                      │
┌─────────────────────────────────────▼───────────────────────────────────────┐
│                            依赖与外部服务                                      │
│  Chat (LLM) │ Reranker │ KnowledgeBase/Chunk/File/MCP/WebSearch/DuckDB       │
└─────────────────────────────────────────────────────────────────────────────┘
```

- **前端**：智能体选择、编辑、对话中的流式展示（思考、工具调用、最终答案）、共享设置。
- **应用层**：HTTP 解析、权限与租户、创建/更新会话与消息、为每次请求创建**独立 EventBus**，订阅后转 SSE。
- **引擎层**：根据 `AgentConfig` 注册工具、构建 SystemPrompt、执行 ReAct 循环并往 EventBus 发事件。
- **依赖层**：LLM、Reranker、知识库/文件/MCP/网络搜索/DuckDB 等由各 Tool 与 Service 使用。

## 3. 核心概念与数据模型

### 3.1 智能体类型与模式

| 类型 | 说明 | 存储与变更 |
|------|------|------------|
| **内置智能体** | 系统预置，如 `builtin-quick-answer`、`builtin-smart-reasoning`、`builtin-data-analyst` | 不落库，由 `types.GetBuiltinAgent(id, tenantID)` 返回；不可删除/不可改基本信息，可复制 |
| **自定义智能体** | 租户下创建 | 表 `custom_agents`，CRUD + 复制；可配置全部 CustomAgentConfig 项 |

| 模式 | 说明 | 入口与执行路径 |
|------|------|----------------|
| `quick-answer` | RAG：检索 → 上下文模板 → 单次 LLM 生成 | 请求走 `agent-chat`，`agentModeEnabled=false` → `executeNormalModeQA` → `KnowledgeQA` |
| `smart-reasoning` | ReAct：多轮 Think → Act → Observe | `agentModeEnabled=true` → `executeAgentModeQA` → `AgentQA` → `CreateAgentEngine` + `engine.Execute` |

`agentModeEnabled` 由 `customAgent.IsAgentMode()` 决定（即 `config.agent_mode == "smart-reasoning"`），请求中的 `AgentEnabled` 仅在没有 customAgent 时使用。

### 3.2 数据模型（核心）

#### CustomAgent（internal/types/custom_agent.go）

- **字段**：`ID`、`Name`、`Description`、`Avatar`、`IsBuiltin`、`TenantID`、`CreatedBy`、`Config CustomAgentConfig`（JSON）、时间戳。
- **表名**：`custom_agents`。

#### CustomAgentConfig（同文件）

持久化配置，主要分组：

- **基础**：`agent_mode`、`system_prompt`、`context_template`
- **模型**：`model_id`、`rerank_model_id`、`temperature`、`max_completion_tokens`、`thinking`
- **Agent 模式**：`max_iterations`、`allowed_tools`、`reflection_enabled`、`mcp_selection_mode`、`mcp_services`
- **Skills**：`skills_selection_mode`、`selected_skills`（仅 smart-reasoning 有效）
- **知识库**：`kb_selection_mode`、`knowledge_bases`、`retrieve_kb_only_when_mentioned`
- **文件限制**：`supported_file_types`（如数据分析师限定 csv/xlsx）
- **FAQ**：`faq_priority_enabled`、`faq_direct_answer_threshold`、`faq_score_boost`
- **网络搜索**：`web_search_enabled`、`web_search_max_results`
- **多轮**：`multi_turn_enabled`、`history_turns`
- **检索**：`embedding_top_k`、`keyword_threshold`、`vector_threshold`、`rerank_top_k`、`rerank_threshold`
- **高级**：`enable_query_expansion`、`enable_rewrite`、`rewrite_prompt_*`、`fallback_strategy`、`fallback_response`、`fallback_prompt`

#### AgentConfig（internal/types/agent.go）

- **用途**：运行时配置，由 Session 层从 CustomAgentConfig 转换而来，供 AgentEngine 使用。
- **关键字段**：除与 CustomAgentConfig 对应项外，包含：
  - `SearchTargets`：预计算检索范围（按租户、知识库、选中文档解析）。
  - `SkillsEnabled`、`SkillDirs`、`AllowedSkills`：Skills 是否启用及白名单。
  - `KnowledgeBases`、`KnowledgeIDs`：当次请求最终使用的知识库/文档 ID 列表（受 `retrieve_kb_only_when_mentioned` 与请求体 @ 提及影响）。

#### 执行状态（同包）

- **AgentState**：`CurrentRound`、`RoundSteps`、`IsComplete`、`FinalAnswer`、`KnowledgeRefs`
- **AgentStep**：`Iteration`、`Thought`、`ToolCalls`、`Timestamp`
- **ToolCall**：`ID`、`Name`、`Args`、`Result`、`Reflection`、`Duration`
- **Tool 接口**：`Name()`、`Description()`、`Parameters()`、`Execute(ctx, args)` → `*ToolResult`

## 4. 请求到执行的完整链路（Smart-Reasoning）

### 4.1 路由与入口

- **路由**：`POST /api/v1/agent-chat/:session_id`（如 router 中 `agentChat.POST("/:session_id", handler.AgentQA)`）。
- **Handler**：`AgentQA` 解析请求（parseQARequest），解析出 session、customAgent、query、knowledgeBaseIDs、knowledgeIDs、summaryModelID 等；根据 `customAgent.IsAgentMode()` 决定：
  - `true` → `executeAgentModeQA(reqCtx)`
  - `false` → `executeNormalModeQA(reqCtx, false)`（走 KnowledgeQA，此处不展开）

### 4.2 executeAgentModeQA 流程

1. **创建用户消息与助手消息**（createUserMessage / createAssistantMessage）。
2. **创建本次请求专属 EventBus**：`setupSSEStream(reqCtx, true)` 内创建 `eventBus := event.NewEventBus()`，并创建 `AgentStreamHandler`，订阅该 EventBus 的各类 Agent 事件，将事件 append 到 `StreamManager`。
3. **异步执行**：`go sessionService.AgentQA(..., streamCtx.eventBus, reqCtx.customAgent, reqCtx.knowledgeBaseIDs, reqCtx.knowledgeIDs)`。
4. **阻塞消费 SSE**：`handleAgentEventsForSSE` 从 StreamManager 取事件推送到 `text/event-stream`，直至收到完成或错误。

因此：**每个 agent-chat 请求拥有独立 EventBus + 独立 AgentStreamHandler 订阅**，避免多会话事件串流。

### 4.3 SessionService.AgentQA 逻辑概要

1. **校验**：customAgent 必填（权限与共享逻辑在 Handler 已做）。
2. **租户与配置**：确定 agent 所属租户（含共享 agent 场景），`customAgent.EnsureDefaults()`，构建运行时 **AgentConfig**：
   - 从 CustomAgentConfig 拷贝：MaxIterations、ReflectionEnabled、Temperature、WebSearch、MCP、Thinking、RetrieveKBOnlyWhenMentioned 等。
   - Skills：`configureSkillsFromAgent` 根据 `skills_selection_mode` / `selected_skills` 解析出 `SkillDirs`、`AllowedSkills`、`SkillsEnabled`。
   - 知识库解析：
     - 若请求带 `knowledgeBaseIDs` / `knowledgeIDs`（@ 提及）：优先使用请求级。
     - 若 `RetrieveKBOnlyWhenMentioned == true` 且无 @ 提及：本请求不使用知识库。
     - 否则：`resolveKnowledgeBasesFromAgent` 按 `kb_selection_mode` 解析出 `KnowledgeBases`。
   - `buildSearchTargets(ctx, agentTenantID, KnowledgeBases, KnowledgeIDs)` 得到 `SearchTargets` 写入 AgentConfig。
3. **模型与上下文**：取 summaryModel、rerankModel（有知识库时）、ContextManager，加载 LLM 历史 `getContextForSession`，多轮关闭时清空历史。
4. **创建引擎**：`agentService.CreateAgentEngine(ctx, agentConfig, summaryModel, rerankModel, eventBus, contextManager, session.ID)`。
5. **执行**：`engine.Execute(ctx, sessionID, assistantMessageID, query, llmContext)`；事件通过 eventBus 发出，由已在 setupSSEStream 中订阅的 AgentStreamHandler 写入 StreamManager，前端通过 SSE 收到。

## 5. Agent 引擎层设计逻辑

### 5.1 CreateAgentEngine（AgentService）

- **校验**：`ValidateConfig(config)`（如 MaxIterations 上限 MAX_ITERATIONS=100）。
- **工具注册**：
  - `toolRegistry := tools.NewToolRegistry()`。
  - `registerTools(ctx, toolRegistry, config, rerankModel, chatModel, sessionID)`：按 `config.AllowedTools`（及无知识库时过滤掉知识库相关工具）、`config.WebSearchEnabled` 注册内置工具；知识库类工具使用 `config.SearchTargets`、rerankModel、chatModel 等。
  - 若配置了 MCP：按 `config.MCPSelectionMode` / `config.MCPServices` 取 MCP 服务列表，`tools.RegisterMCPTools(..., toolRegistry, enabledServices, mcpManager)`。
- **知识库与选中文档**：`getKnowledgeBaseInfos`、`getSelectedDocumentInfos` 得到 `KnowledgeBaseInfo`、`SelectedDocumentInfo` 供 SystemPrompt 使用。
- **系统提示词**：若 `config.UseCustomSystemPrompt`，则 `systemPromptTemplate = config.ResolveSystemPrompt(config.WebSearchEnabled)`。
- **创建引擎**：`agent.NewAgentEngine(config, chatModel, toolRegistry, eventBus, kbInfos, selectedDocs, contextManager, sessionID, systemPromptTemplate)`。
- **Skills**：若 `config.SkillsEnabled && len(config.SkillDirs) > 0`，则 `initializeSkillsManager`（按环境变量选择沙箱 docker/local/disabled），注册 `read_skill`、`execute_skill_script`（沙箱非 disabled 时），并 `engine.SetSkillsManager(skillsManager)`。

### 5.2 AgentEngine 结构

- **持有**：`config *AgentConfig`、`toolRegistry`、`chatModel`、`eventBus`、`knowledgeBasesInfo`、`selectedDocs`、`contextManager`、`sessionID`、`systemPromptTemplate`、可选 `skillsManager`。
- **Execute 流程概要**：
  - 初始化 `AgentState`（RoundSteps、KnowledgeRefs、IsComplete=false）。
  - 构建 SystemPrompt：若启用 Skills 则带 Skills 元数据（Level 1 Progressive Disclosure），否则走默认 BuildSystemPrompt；占位符如 `{{knowledge_bases}}`、`{{web_search_status}}`、`{{current_time}}` 在此或调用处替换。
  - `buildMessagesWithLLMContext(systemPrompt, query, llmContext)` 得到首轮 messages。
  - `buildToolsForLLM()` 从 toolRegistry 取所有已注册工具的 FunctionDefinitions，供 LLM function calling。
  - 进入 **executeLoop**，结束后若有需要则写回最终答案与引用到 contextManager，并 `toolRegistry.Cleanup(ctx)`（如 DuckDB 清理）。

### 5.3 ReAct 循环（executeLoop）

每轮迭代（直到 `state.CurrentRound >= config.MaxIterations` 或提前结束）：

1. **Think**  
   `streamThinkingToEventBus(ctx, messages, tools, state.CurrentRound, sessionID)`：  
   调用 LLM（带 function calling、thinking 等），流式输出思考内容，通过 EventBus 发送 `EventAgentThought`；返回完整 content + tool_calls。

2. **判断结束**  
   若 `response.FinishReason == "stop"` 且 `len(response.ToolCalls) == 0`：  
   视为最终答案，写入 `state.FinalAnswer`、`state.IsComplete = true`，发送 `EventAgentFinalAnswer`（Done）、然后在本轮末尾发送 `EventAgentComplete`，退出循环。

3. **Act**  
   若有 `response.ToolCalls`：  
   对每个 tool call：发送 `EventAgentToolCall` → `toolRegistry.ExecuteTool` → 发送 `EventAgentToolResult`（及可选的 `EventAgentTool` 内部监控）；若开启 Reflection，则 `streamReflectionToEventBus` 并写回对应 ToolCall.Reflection。

4. **Observe**  
   `appendToolResults(ctx, messages, step)`：将本轮的 assistant message（含 thought + tool_calls）与各 tool 的 role=tool 消息追加到 messages，并写入 contextManager，供下一轮与持久化。

5. **下一轮**  
   `state.CurrentRound++`，继续循环。

若达到 `MaxIterations` 仍未得到最终答案：调用 `streamFinalAnswerToEventBus` 用 LLM 根据已有步骤合成最终答案，再发 `EventAgentComplete`。

### 5.4 事件类型（internal/event）

与 Agent 流式相关的主要事件：

- `EventAgentThought`：思考过程（流式块）。
- `EventAgentToolCall`：工具调用（名称、参数、迭代）。
- `EventAgentToolResult`：工具结果（输出、错误、耗时、结构化 Data）。
- `EventAgentReflection`：反思内容（若开启）。
- `EventAgentReferences`：知识引用。
- `EventAgentFinalAnswer`：最终答案（流式块，Done 表示结束）。
- `EventAgentComplete`：本轮 Agent 执行完成（含 FinalAnswer、KnowledgeRefs、AgentSteps、MessageID 等）。
- `EventError`：错误信息。

AgentStreamHandler 订阅上述事件，将事件 append 到 StreamManager，由 SSE 推送到前端；前端 AgentStreamDisplay 等组件按类型渲染思考、工具调用、结果、最终答案。

## 6. 工具体系与扩展机制

### 6.1 工具名与默认白名单（internal/agent/tools/definitions.go）

- **常量**：thinking、todo_write、knowledge_search、grep_chunks、list_knowledge_chunks、query_knowledge_graph、get_document_info、database_query、data_analysis、data_schema、web_search、web_fetch、read_skill、execute_skill_script。
- **DefaultAllowedTools()**：默认允许的列表（不含 web_search/web_fetch/read_skill/execute_skill_script；后两者由 Skills 启用时注册）。
- **AvailableToolDefinitions()**：供 UI/设置使用的工具元数据（name、label、description）。

### 6.2 注册逻辑（AgentService.registerTools）

- 使用 `config.AllowedTools`，若为空则 `DefaultAllowedTools()`。
- **无知识库时**：过滤掉知识库/数据类工具（如 knowledge_search、grep_chunks、list_knowledge_chunks、query_knowledge_graph、get_document_info、database_query、data_analysis、data_schema）；若无知识库且未开 web_search，还会去掉 todo_write。
- **WebSearchEnabled**：追加 web_search、web_fetch。
- 按工具名 switch 创建对应 Tool 实现并 `registry.RegisterTool(tool)`；同名已存在则不再覆盖（first-wins，防劫持）。

### 6.3 MCP 与 Skills

- **MCP**：CreateAgentEngine 内根据 tenantID、config.MCPSelectionMode、config.MCPServices 列出 MCP 服务，只注册启用且允许的服务下的工具到同一 ToolRegistry。
- **Skills**：仅在 `config.SkillsEnabled && len(config.SkillDirs) > 0` 时初始化 Skills Manager，扫描目录、校验 YAML frontmatter、注册 read_skill 与（沙箱非 disabled 时）execute_skill_script；SystemPrompt 中注入 Skills 元数据（Progressive Disclosure Level 1）。

### 6.4 ToolRegistry（internal/agent/tools/registry.go）

- **RegisterTool**：按 `tool.Name()` 存入 map，已存在则忽略。
- **GetTool**、**ListTools**、**GetFunctionDefinitions**、**ExecuteTool**：按名查找并执行。
- **Cleanup**：对实现了清理逻辑的工具（如 DataAnalysis 的 DuckDB）做清理。

### 7.1 API（frontend/src/api/agent/index.ts）

- 与后端一致：CustomAgent / CustomAgentConfig 类型、agents CRUD、placeholders、copy。

- **共享**：AgentShareSettings、组织/租户下 agent-shares、shared-agents 等接口与页面。

## 8. 配置与安全要点

- **迭代上限**：MaxIterations 在 ValidateConfig 中限制 ≤ MAX_ITERATIONS（100），防止死循环。
- **工具白名单**：仅 `config.AllowedTools` 中出现的工具会被注册并暴露给 LLM；无知识库时再过滤知识库相关工具。
- **知识库/文档范围**：由 `SearchTargets` 与请求/agent 的 `KnowledgeBases`、`KnowledgeIDs` 严格限定检索与数据访问范围；共享 agent 时使用 agent 所属租户做检索范围。
- **Skills 沙箱**：脚本执行通过 sandbox.Manager（docker/local/disabled），执行前有脚本与参数安全校验（见 docs/agent-skills.md）。

| 扩展点 | 位置 | 方式 |
|--------|------|------|
| 新内置智能体 | `types.BuiltinAgentRegistry`、`builtinAgentIDsOrdered` | 增加 ID 与工厂函数，必要时扩展 GetBuiltinAgentIDs |
