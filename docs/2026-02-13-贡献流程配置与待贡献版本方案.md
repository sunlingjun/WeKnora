## 二、远程仓库配置调整（一次性）

在项目根目录执行：

```powershell
# 1. 将当前 origin（Tencent）改名为 upstream
git remote rename origin upstream

# 2. 添加你的 Fork 为 origin
git remote add origin https://github.com/sunlingjun/WeKnora.git

# 3. 验证
git remote -v
```

**预期输出：**

```
origin    https://github.com/sunlingjun/WeKnora.git (fetch)
origin    https://github.com/sunlingjun/WeKnora.git (push)
upstream  https://github.com/Tencent/WeKnora.git (fetch)
upstream  https://github.com/Tencent/WeKnora.git (push)
```

**若本地已有大量提交尚未推送到任何远程：**

- 若之前是直接 clone 的 Tencent/WeKnora，可先 `git push upstream main` 备份到上游（需有权限）或先推送到 Fork：  
  `git push origin main`（需先完成步骤 2，再执行一次 `git push -u origin main`）。
- 推荐：先完成上述 3 步，再执行一次同步（见下节），然后把本地分支推送到 `origin`。

## 三、贡献流程中的角色

| 操作 | 使用 |
|------|------|
| 拉取/同步最新代码 | `git fetch upstream`、`git merge upstream/main` |
| 推送你的功能分支、提交 PR | `git push origin feature/xxx` |
| 克隆与日常拉取 | `git clone origin`、`git pull origin main` |

**日常同步（保持 Fork 与上游一致）：**

```powershell
git checkout main
git fetch upstream
git merge upstream/main
git push origin main
```

之后在**功能分支**上开发，最后推送到 `origin` 并在 GitHub 上向 **Tencent/WeKnora** 提 PR。

### 版本 1：自定义 SVG 组件设计

**依据**：[2026-02-12-自定义SVG组件化方案](./2026-02-12-自定义SVG组件化方案.md)

| 项 | 内容 |
|----|------|
| **分支名** | `feature/svg-icon-component` |
| **类型** | 新功能（feat） |
| **范围** | 前端图标体系 |

**1. 目标摘要**

- 自定义 SVG 图标组件化，统一引用与主题适配（含 TDesign 主题）。
- 用 `currentColor` 替代多份同图标不同色文件（如 `xxx.svg` / `xxx-green.svg`）。

**2. 实施步骤（建议提交粒度）**

| 步骤 | 内容 | 建议 commit |
|------|------|-------------|
| Phase 1 | 新增 `frontend/src/components/icons/`：`SvgIcon.vue`、`registry.ts`、`index.ts`；规范化首批 SVG（fill/stroke → currentColor） | `feat(frontend): add SvgIcon component and icon registry` |
| Phase 2 | 将 `<img src="xxx.svg">` 替换为 `<SvgIcon name="xxx" />`，删除冗余变体（如 -green/-grey） | `refactor(frontend): replace img SVG with SvgIcon in menu/settings` 等（可按页面拆分） |
| Phase 3 | 补充其余图标、`aria-label`、`role="img"` 等无障碍与规范 | `feat(frontend): complete SvgIcon usage and a11y` |

**3. PR 要点**

- **标题示例**：`feat(frontend): 自定义 SVG 组件化与主题适配`
- **描述建议**：说明目标（统一管理、主题适配、去冗余）、目录结构、`SvgIcon` 的 props（name/size/color/theme）、与 TDesign CSS 变量的对应关系。
- **测试**：亮/暗主题切换下图标颜色正确；关键页面（如菜单、设置）无缺失图标。

**4. 可能冲突**

- 与上游改动重叠的文件：`menu.vue`、`Input-field.vue`、各设置页。提交前先按「日常同步」拉齐 `upstream/main`，在功能分支上合并后再改。

### 版本 2：统一颜色升级

**依据**：[2026-02-05-颜色统一化改造](./2026-02-05-颜色统一化改造.md)

| 项 | 内容 |
|----|------|
| **分支名** | `feature/color-unify-tdesign-vars` |
| **类型** | 样式/重构（style + refactor） |
| **范围** | 前端全局颜色与主题变量 |

**1. 目标摘要**

- 将硬编码颜色替换为 TDesign CSS 变量（如 `--td-text-color-primary`、`--td-brand-color` 等）。
- 支持暗色模式，保留 fallback（如 `var(--td-brand-color, #07c05f)`）。

**2. 实施步骤（你处已完成，贡献时建议整理为）**

| 步骤 | 内容 | 建议 commit |
|------|------|-------------|
| 1 | 在 `theme.css`（或项目主题入口）中确认/补充 TDesign 变量与 fallback | `style(frontend): align theme CSS variables with TDesign` |
| 2 | 按模块分批替换：Settings → Components → Knowledge → Chat/Tool 等（见颜色改造文档「已完成组件」） | `style(frontend): replace hardcoded colors in Settings/Components` 等 |
| 3 | 统一边框/背景/功能色（见文档 2.2–2.5 映射表），保留文档中「保留的特殊情况」 | `style(frontend): unify border and semantic colors` |

**3. PR 要点**

- **标题示例**：`style(frontend): 统一颜色为 TDesign CSS 变量并支持暗色模式`
- **描述建议**：贴「颜色映射对照表」摘要、说明 fallback 策略、列出已改动的组件/目录；注明保留项（如 Agent 模式特殊色、AgentAvatar 渐变）。
- **测试**：亮/暗主题切换、关键页面无颜色错乱；可附 `grep -r "var(--td-" frontend/src | wc -l` 等说明变量使用范围。

**4. 可能冲突**

- `theme.css`、`menu.vue`、`Input-field.vue`、`AgentSelector.vue` 等与上游共有改动，需在分支上先同步 `upstream/main` 再合并或三路合并（可参考 [2026-02-11-项目升级与合并指南](./2026-02-11-项目升级与合并指南.md)）。

## 五、本地大量修改如何划分到细化分支

当本地已有大量修改（颜色 + SVG 混在一起）时，可按下面方式拆成「版本 2：颜色」和「版本 1：SVG」两条分支，分别提 PR。

### 5.1 文件归属清单（便于按版本拣选）

| 归属 | 说明 | 路径/文件示例 |
|------|------|----------------|
| **仅颜色（版本 2）** | 只含 CSS 变量替换、无 SvgIcon/图标引用 | `frontend/src/assets/theme/theme.css`，以及仅改颜色的 .vue/.less |
| **仅 SVG（版本 1）** | 新增图标组件或只改图标引用 | `frontend/src/components/icons/` 整个目录（SvgIcon.vue、registry.ts、index.ts），`frontend/src/assets/img/*.svg` 的增删 |
| **重叠** | 同一文件里既有颜色变量又有 SvgIcon/img 改动 | `menu.vue`、`Input-field.vue`、`AgentSelector.vue`、`KnowledgeBaseList.vue`、`MentionSelector.vue`、各 Settings 与 Knowledge 相关组件等 |

重叠文件需要按「块」区分：颜色相关块归版本 2，SvgIcon/`<img *.svg>` 相关块归版本 1。

### 5.2 场景 A：所有改动都还未提交（工作区脏）

思路：先备份当前状态，再从「干净基线」拉出两条分支，用 **`git add` 按文件/按块** 分别只提交颜色或只提交 SVG。

```powershell
# 1. 备份当前全部修改（含未跟踪文件）
git stash push -u -m "all-local-color-and-svg"

# 2. 确保 main 与上游一致（或已是你想要的基线）
git checkout main
git fetch upstream
git merge upstream/main

# 3. 先做「版本 2：颜色」分支
git checkout -b feature/color-unify-tdesign-vars
git stash pop

# 4. 只暂存「颜色相关」的改动
# 4a. 先加仅颜色文件（如 theme.css）
git add frontend/src/assets/theme/theme.css

# 4b. 再按需添加其他仅颜色组件（见 5.1 清单；重叠文件用 add -p 只选颜色相关块）
git add frontend/src/views/settings/
git add frontend/src/views/chat/components/
# … 其他仅颜色或你确认整文件都是颜色改动的路径

# 4c. 重叠文件：用交互式暂存，只选颜色相关的 hunk
git add -p frontend/src/components/menu.vue
git add -p frontend/src/components/Input-field.vue
# … 对每个重叠文件，在提示时选 y（暂存）/ n（不暂存）/ s（拆分块）

# 5. 提交并推送颜色分支
git commit -m "style(frontend): unify colors with TDesign CSS variables"
git push origin feature/color-unify-tdesign-vars

# 6. 剩余改动还在工作区；再建「版本 1：SVG」分支
git stash push -u -m "remaining-svg"
git checkout main
git checkout -b feature/svg-icon-component
git stash pop

# 7. 只暂存 SVG 相关：icons 目录 + 重叠文件中 SvgIcon/img 相关块
git add frontend/src/components/icons/
git add -p frontend/src/components/menu.vue
git add -p frontend/src/components/Input-field.vue
# … 其他引用 SvgIcon 或 img *.svg 的文件

# 8. 提交并推送 SVG 分支
git commit -m "feat(frontend): add SvgIcon component and replace img SVG usage"
git push origin feature/svg-icon-component
```

**注意**：`git add -p` 时，颜色相关块（如 `var(--td-xxx)`）选 `y`，SvgIcon/`<img` 相关块选 `n`（在颜色分支）；在 SVG 分支里反过来。

### 5.3 场景 B：所有改动已在 main 上（一个或多个 commit）

思路：用**备份分支**保留当前 main，再在两条功能分支上分别「拣选」只属于该版本的文件。

```powershell
# 1. 备份当前 main（含你所有提交）
git branch backup-all-$(Get-Date -Format "yyyyMMdd")

# 2. 从当前 main 拉出颜色分支
git checkout -b feature/color-unify-tdesign-vars

# 3. 软重置，使改动回到工作区，再只加颜色相关并提交
# 若只有 1 个大 commit：git reset --soft HEAD~1
# 若有 N 个 commit 要一起拆：git reset --soft HEAD~N 或 git reset --soft upstream/main
git reset --soft HEAD~1
# 然后按 5.2 步骤 4：只 add 颜色相关文件/块
git add frontend/src/assets/theme/theme.css
# … 其他颜色文件；重叠文件用 git add -p
git commit -m "style(frontend): unify colors with TDesign CSS variables"
git push origin feature/color-unify-tdesign-vars

# 4. 从备份拉出 SVG 分支，只保留 SVG 相关（分支名与步骤 1 一致，若用了日期则改为 backup-all-yyyyMMdd）
git checkout backup-all
git checkout -b feature/svg-icon-component
git reset --soft HEAD~1
git add frontend/src/components/icons/
# 重叠文件用 add -p 只选 SVG 相关块
git add -p frontend/src/components/menu.vue
# ...
git commit -m "feat(frontend): add SvgIcon component and replace img SVG usage"
git push origin feature/svg-icon-component
```

（备份分支名中的日期按你执行日替换；若只有一个备份可写 `git branch backup-all`。）

### 5.4 场景 C：部分已提交、部分未提交

1. 先把**未提交**的改动 `git stash push -u -m "mixed"`。
2. 对**已提交**部分：按 5.3 用 backup 分支 + 在两个 feature 分支上分别 `reset --soft`，只保留对应版本的改动。
3. 对 **stash** 里的改动：`stash pop` 到对应 feature 分支，用 `git add` / `git add -p` 只选该版本需要的文件或块，再 commit。

### 5.5 小结

| 场景 | 关键操作 |
|------|----------|
| 全未提交 | `stash` 备份 → 两个分支上分别 `stash pop` + 按版本 `add` / `add -p` |
| 全已提交 | `backup` 分支 → 两分支上 `reset --soft`，再按版本 `add` / `add -p` |
| 混合 | stash 未提交 + 已提交按 5.3 拆，再在对应分支上应用 stash 并 `add` |

**重叠文件**：用 `git add -p <文件>` 按块（hunk）选择「颜色」或「SVG」归属，避免同一文件混在两个 PR 里。

## 六、两版本的顺序与依赖

| 顺序 | 版本 | 建议 |
|------|------|------|
| 先 | **版本 2：统一颜色升级** | 不依赖 SVG 组件，且你处已完成，可先整理为单一分支、单 PR，快速贡献。 |
| 后 | **版本 1：自定义 SVG 组件设计** | 可与「颜色统一」共用同一套 TDesign 变量；若版本 2 已合入上游，再提 SVG 组件 PR 冲突更少。 |

若希望**两个版本都基于当前本地状态**一起贡献，也可以：

- 分支 A：`feature/color-unify-tdesign-vars`（仅颜色相关提交）
- 分支 B：`feature/svg-icon-component`（在 A 之上或从 `upstream/main` 拉取后再做 SVG，视上游是否已合入 A）

## 七、快速检查清单（提交 PR 前）

- [ ] 已执行「二、远程仓库配置调整」且 `git remote -v` 正确
- [ ] 已从 `upstream/main` 同步：`git fetch upstream`、`git merge upstream/main`
- [ ] 在功能分支上开发：`feature/xxx`，未在 `main` 直接提交
- [ ] 提交信息符合 Conventional Commits：`feat:` / `style:` / `refactor:` 等
- [ ] 前端：`cd frontend && npm run lint && npm run type-check`
- [ ] 若为颜色/SVG：本地切换亮/暗主题、检查关键页面
