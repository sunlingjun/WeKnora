# 密码加盐规则与导入问题排查

## 一、密码加密规则

### 1.1 加密算法

项目使用 **bcrypt** 算法进行密码加密，具体实现如下：

```go
// 密码加密
hashedPassword, err := bcrypt.GenerateFromPassword([]byte(req.Password), bcrypt.DefaultCost)

// 密码验证
err = bcrypt.CompareHashAndPassword([]byte(user.PasswordHash), []byte(req.Password))
```

**代码位置**：`internal/application/service/user.go`

### 1.2 bcrypt 特点

1. **自动加盐**：bcrypt 每次加密都会自动生成随机盐（salt），即使相同的密码，每次加密结果都不同
2. **盐值包含在哈希中**：bcrypt 的哈希值格式为 `$2a$10$...`，其中包含了盐值和成本参数
3. **成本参数**：使用 `bcrypt.DefaultCost`（值为 10），表示迭代次数为 2^10 = 1024 次
4. **哈希长度**：bcrypt 哈希值固定为 **60 个字符**

### 1.3 bcrypt 哈希值格式

```
$2a$10$N9qo8uLOickgx2ZMRZoMyeIjZAgcfl7p92ldGxad68LJZdL17lhWy
│  │  │  │                    │
│  │  │  │                    └─ 哈希值（31 字符）
│  │  │  └─ 盐值（22 字符）
│  │  └─ 成本参数（10 = 2^10 次迭代）
│  └─ 版本标识（2a, 2b, 2y）
└─ 算法标识
```

**完整长度**：60 字符

### 1.4 数据库字段定义

```sql
password_hash VARCHAR(255) NOT NULL
```

**字段长度**：255 字符（足够存储 bcrypt 哈希值）

## 二、导入后验证失败的原因分析

### 2.1 常见原因

#### 🔴 原因 1：字段长度截断（最常见）

**问题**：虽然字段定义为 `VARCHAR(255)`，但在某些导入工具或 SQL 客户端中，如果数据被截断，会导致验证失败。

**检查方法**：
```sql
-- 检查密码哈希长度
SELECT 
    id,
    email,
    LENGTH(password_hash) as hash_length,
    password_hash
FROM users
WHERE email = 'your-email@example.com';

-- bcrypt 哈希值应该是 60 字符
-- 如果长度不是 60，说明被截断了
```

**解决方案**：
```sql
-- 确保字段长度足够
ALTER TABLE users ALTER COLUMN password_hash TYPE VARCHAR(255);

-- 或者使用 TEXT 类型（推荐）
ALTER TABLE users ALTER COLUMN password_hash TYPE TEXT;
```

#### 🟡 原因 2：字符编码问题

**问题**：bcrypt 哈希值包含特殊字符（`$`），如果数据库字符集不正确，可能导致字符被转换或损坏。

**检查方法**：
```sql
-- 检查数据库字符集
SHOW SERVER_ENCODING;
-- 应该是 UTF8 或 UTF8MB4

-- 检查表的字符集
SELECT 
    table_name,
    table_encoding
FROM pg_tables 
WHERE table_name = 'users';

-- 检查字段字符集
SELECT 
    column_name,
    data_type,
    character_set_name
FROM information_schema.columns
WHERE table_name = 'users' AND column_name = 'password_hash';
```

**解决方案**：
```sql
-- 确保数据库使用 UTF8 编码
CREATE DATABASE weknora WITH ENCODING 'UTF8';

-- 或者修改现有数据库
ALTER DATABASE weknora SET ENCODING 'UTF8';
```

#### 🟡 原因 3：导入时特殊字符转义问题

**问题**：使用 SQL 导入时，`$` 字符可能需要转义，或者导入工具自动转义导致数据损坏。

**检查方法**：
```sql
-- 检查密码哈希值格式
SELECT 
    email,
    password_hash,
    SUBSTRING(password_hash, 1, 7) as prefix
FROM users
WHERE email = 'your-email@example.com';

-- 前缀应该是 $2a$10$ 或 $2b$10$
-- 如果前缀不对，说明导入时有问题
```

**解决方案**：
```sql
-- 使用正确的转义方式导入
-- PostgreSQL
INSERT INTO users (id, email, password_hash, ...) 
VALUES ('...', '...', '$2a$10$...', ...);

-- 或者使用 COPY 命令（推荐）
COPY users (id, email, password_hash, ...) 
FROM '/path/to/data.csv' 
WITH (FORMAT csv, HEADER true);
```

#### 🟢 原因 4：字符串前后空格

**问题**：导入时可能在密码哈希值前后添加了空格。

**检查方法**：
```sql
-- 检查是否有前后空格
SELECT 
    email,
    LENGTH(password_hash) as length,
    LENGTH(TRIM(password_hash)) as trimmed_length,
    password_hash
FROM users
WHERE email = 'your-email@example.com';

-- 如果 length != trimmed_length，说明有空格
```

**解决方案**：
```sql
-- 清理前后空格
UPDATE users 
SET password_hash = TRIM(password_hash)
WHERE LENGTH(password_hash) != LENGTH(TRIM(password_hash));
```

#### 🟢 原因 5：换行符或控制字符

**问题**：导入时可能引入了换行符（`\n`）或其他控制字符。

**检查方法**：
```sql
-- 检查是否有换行符
SELECT 
    email,
    password_hash,
    POSITION(E'\n' IN password_hash) as has_newline,
    POSITION(E'\r' IN password_hash) as has_carriage_return
FROM users
WHERE email = 'your-email@example.com';
```

**解决方案**：
```sql
-- 清理换行符和控制字符
UPDATE users 
SET password_hash = REGEXP_REPLACE(password_hash, E'[\\n\\r\\t]', '', 'g')
WHERE password_hash ~ E'[\\n\\r\\t]';
```

## 三、正确的导入方法

### 3.1 使用 pg_dump/pg_restore（推荐）

```bash
# 从本地库导出
pg_dump -h localhost -U username -d local_db -t users --data-only > users_data.sql

# 导入到测试库
psql -h test_host -U username -d test_db -f users_data.sql
```

**优点**：
- ✅ 保持数据完整性
- ✅ 自动处理特殊字符
- ✅ 保持字符编码

### 3.2 使用 COPY 命令

```sql
-- 导出
COPY (
    SELECT id, email, password_hash, username, tenant_id, is_active, created_at, updated_at
    FROM users
) TO '/tmp/users.csv' WITH (FORMAT csv, HEADER true);

-- 导入
COPY users (id, email, password_hash, username, tenant_id, is_active, created_at, updated_at)
FROM '/tmp/users.csv' WITH (FORMAT csv, HEADER true);
```

### 3.3 使用 INSERT 语句（小数据量）


## 四、验证导入的数据

### 4.1 检查密码哈希格式

```sql
-- 创建验证函数
CREATE OR REPLACE FUNCTION validate_password_hash(hash TEXT)
RETURNS BOOLEAN AS $$
BEGIN
    -- 检查长度
    IF LENGTH(hash) != 60 THEN
        RETURN FALSE;
    END IF;
    
    -- 检查前缀
    IF hash NOT LIKE '$2a$10$%' AND hash NOT LIKE '$2b$10$%' AND hash NOT LIKE '$2y$10$%' THEN
        RETURN FALSE;
    END IF;
    
    -- 检查字符集（只包含字母、数字、$、/、.）
    IF hash !~ '^[$./A-Za-z0-9]{60}$' THEN
        RETURN FALSE;
    END IF;
    
    RETURN TRUE;
END;
$$ LANGUAGE plpgsql;

-- 检查所有用户的密码哈希
SELECT 
    id,
    email,
    password_hash,
    LENGTH(password_hash) as length,
    validate_password_hash(password_hash) as is_valid
FROM users
WHERE validate_password_hash(password_hash) = FALSE;
```

### 4.2 修复无效的密码哈希

如果发现密码哈希无效，需要重新设置密码：

```sql
-- 方法1：重置密码（用户需要重新设置）
UPDATE users 
SET password_hash = NULL 
WHERE validate_password_hash(password_hash) = FALSE;

-- 方法2：设置临时密码（需要知道原始密码）
-- 使用 Go 程序生成新的哈希值，然后更新
```

## 五、使用 Go 程序验证密码

### 5.1 创建验证脚本

创建文件 `scripts/verify_password.go`：

```go
package main

import (
    "fmt"
    "os"
    "golang.org/x/crypto/bcrypt"
)

func main() {
    if len(os.Args) != 3 {
        fmt.Println("Usage: go run verify_password.go <password_hash> <password>")
        os.Exit(1)
    }
    
    hash := os.Args[1]
    password := os.Args[2]
    
    err := bcrypt.CompareHashAndPassword([]byte(hash), []byte(password))
    if err != nil {
        fmt.Printf("❌ 密码验证失败: %v\n", err)
        os.Exit(1)
    }
    
    fmt.Println("✅ 密码验证成功")
}
```

### 5.2 使用方法

```bash
# 编译
go build -o verify_password scripts/verify_password.go

# 验证
./verify_password '$2a$10$...' 'your-password'
```

## 七、常见问题 FAQ

### Q1: bcrypt 哈希值可以跨数据库使用吗？

**A**: 可以。bcrypt 哈希值是自包含的（包含盐值），可以在任何支持 bcrypt 的系统间迁移。

### Q2: 为什么相同的密码每次加密结果不同？

**A**: 因为 bcrypt 每次加密都会生成随机盐值，这是正常的安全特性。

### Q3: 导入后密码验证失败，但哈希值看起来正确？

**A**: 检查以下几点：
1. 哈希值长度是否为 60 字符
2. 是否有前后空格或换行符
3. 字符编码是否正确（UTF8）
4. 特殊字符是否被正确转义

### Q4: 可以使用 TEXT 类型存储密码哈希吗？

**A**: 可以。虽然 VARCHAR(255) 足够，但使用 TEXT 更安全，避免长度限制问题。

### Q5: 如何批量重置用户密码？

**A**: 
```sql
-- 设置所有用户密码为临时密码（需要用户首次登录后修改）
-- 注意：这需要知道临时密码的 bcrypt 哈希值
UPDATE users 
SET password_hash = '$2a$10$...'  -- 临时密码的哈希值
WHERE ...;
```
