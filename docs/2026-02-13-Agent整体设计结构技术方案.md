# Agent 整体设计结构技术方案

## 1. 概述

WeKnora 的 Agent 模块提供两种运行模式：**快速问答（quick-answer）** 与 **智能推理（smart-reasoning）**。快速问答基于 RAG 直接检索生成回答；智能推理采用 **ReAct（Reasoning + Acting）** 框架，支持多步思考、工具调用、反思与 MCP/Skills 扩展，通过 EventBus 流式输出到前端。

### 1.1 设计目标

- **双模式统一**：同一套「智能体」抽象下支持 RAG 与 ReAct，通过 `agent_mode` 与配置区分。
- **可配置智能体**：内置智能体 + 自定义智能体（CustomAgent），支持知识库选择、工具白名单、系统提示词等。
- **工具可扩展**：内置工具 + MCP 工具 + Agent Skills，按需注册、白名单控制。
- **流式与可观测**：思考、工具调用、结果、最终答案均通过 EventBus 推送，便于 SSE 与追踪。

## 3. 核心概念与数据模型

### 3.1 智能体类型与模式

| 类型         | 说明 |
|--------------|------|
| **内置智能体** | 系统预置，不可删除/不可改基本信息，可复制。如 `builtin-quick-answer`、`builtin-smart-reasoning`、`builtin-data-analyst`。 |
| **自定义智能体** | 租户下创建，CRUD + 复制，配置灵活。 |

| 模式               | 说明 |
|--------------------|------|
| `quick-answer`     | RAG 模式：检索 → 上下文模板 → 单次 LLM 生成，无 ReAct 循环。 |
| `smart-reasoning`  | ReAct 模式：多轮 Think → Act（工具调用）→ Observe，直至 LLM 返回 stop 或无工具调用。 |

### 3.2 数据模型（核心）

**CustomAgent**（`internal/types/custom_agent.go`）

- `ID`、`Name`、`Description`、`Avatar`、`IsBuiltin`、`TenantID`、`CreatedBy`
- `Config: CustomAgentConfig`（JSON 存储）

**CustomAgentConfig** 主要分组：

- **基础**：`agent_mode`、`system_prompt`、`context_template`
- **模型**：`model_id`、`rerank_model_id`、`temperature`、`max_completion_tokens`、`thinking`
- **Agent 模式**：`max_iterations`、`allowed_tools`、`reflection_enabled`、`mcp_selection_mode`、`mcp_services`
- **Skills**：`skills_selection_mode`、`selected_skills`（仅 smart-reasoning）
- **知识库**：`kb_selection_mode`、`knowledge_bases`、`retrieve_kb_only_when_mentioned`
- **文件限制**：`supported_file_types`（如数据分析师限定 csv/xlsx）
- **FAQ**：`faq_priority_enabled`、`faq_direct_answer_threshold`、`faq_score_boost`
- **网络搜索**：`web_search_enabled`、`web_search_max_results`
- **多轮**：`multi_turn_enabled`、`history_turns`
- **检索**：`embedding_top_k`、`keyword_threshold`、`vector_threshold`、`rerank_top_k`、`rerank_threshold`
- **高级**：`enable_query_expansion`、`enable_rewrite`、`rewrite_prompt_*`、`fallback_strategy`、`fallback_response`、`fallback_prompt`

**AgentConfig**（`internal/types/agent.go`）

- 运行时从 CustomAgentConfig 转换而来，供 AgentEngine 使用。
- 包含 `SearchTargets`（预计算检索范围）、`SkillsEnabled`、`SkillDirs`、`AllowedSkills` 等。

**Agent 执行状态**（同包）

- `AgentState`：`CurrentRound`、`RoundSteps`、`IsComplete`、`FinalAnswer`、`KnowledgeRefs`
- `AgentStep`：`Iteration`、`Thought`、`ToolCalls`、`Timestamp`
- `ToolCall`：`ID`、`Name`、`Args`、`Result`、`Reflection`、`Duration`
- `Tool` 接口：`Name()`、`Description()`、`Parameters()`、`Execute(ctx, args)`

## 4. 后端分层结构

### 4.1 接口与实现

- **interfaces**：`AgentService`、`AgentEngine`、`ContextManager`、`StreamManager` 等。
- **AgentService**（`internal/application/service/agent_service.go`）：
  - `CreateAgentEngine(ctx, config, chatModel, rerankModel, eventBus, contextManager, sessionID)`：校验配置、创建 ToolRegistry、按 config 注册内置工具、按 MCP 配置注册 MCP 工具、拉取知识库/选中文档信息、若开启 Skills 则初始化 Skills Manager 并挂到引擎上，最后返回 `AgentEngine`。
  - `registerTools`：根据 `allowed_tools`、是否有知识库、是否开启 web_search 等决定注册哪些工具；知识库相关工具依赖 `SearchTargets`、Reranker、Chat 等。
  - `initializeSkillsManager`：根据环境变量选择沙箱（docker/local/disabled），创建 `skills.Manager`，注册 `read_skill` 与（沙箱非 disabled 时）`execute_skill_script`。

### 4.2 Agent 引擎（internal/agent）

- **engine.go**
  - `AgentEngine`：持有一轮执行所需的 `AgentConfig`、`ToolRegistry`、`Chat`、`EventBus`、知识库/选中文档信息、`ContextManager`、`sessionID`、系统提示词模板、可选 `skills.Manager`。
  - `Execute(ctx, sessionID, messageID, query, llmContext)`：  
    初始化 `AgentState`；构建 SystemPrompt（含可选 Skills 元数据）；若有历史则写入 ContextManager；进入 `executeLoop`；结束后写回最终答案与引用到 ContextManager；返回 `AgentState`。
  - **ReAct 循环**（`executeLoop`）：
    1. **Think**：`streamThinkingToEventBus` 调用 LLM（function calling + 流式思考），发出 `EventAgentThought`。
    2. 若 `finish_reason == "stop"` 且无 tool_calls → 视为最终答案，发 `EventAgentFinalAnswer`（Done）、`EventAgentComplete`，结束。
    3. **Act**：遍历 `response.ToolCalls`，发 `EventAgentToolCall`，`toolRegistry.ExecuteTool`，发 `EventAgentToolResult`；若有反思则发 `EventAgentReflection`；将 Observation 写入 messages。
    4. 可选 **Reflection**：若开启反思，再调 LLM 生成反思并追加到 messages。
    5. 步数、引用等写入 state，进入下一轮，直到达到 `max_iterations` 或提前结束。

- **prompts.go**
  - 根据是否挂载知识库、是否启用网络搜索、是否启用 Skills 等，组装 SystemPrompt。
  - 占位符如 `{{web_search_status}}`、`{{current_time}}` 等在此或调用处替换。

### 4.3 工具体系（internal/agent/tools）

- **registry.go**：`ToolRegistry` 注册、按名获取、`GetFunctionDefinitions`、`ExecuteTool`、`Cleanup`（如 DataAnalysis 的 DuckDB 清理）。
- **definitions.go**：工具名常量（如 `thinking`、`todo_write`、`knowledge_search`、`grep_chunks`、`list_knowledge_chunks`、`query_knowledge_graph`、`get_document_info`、`database_query`、`data_schema`、`data_analysis`、`web_search`、`web_fetch`、`read_skill`、`execute_skill_script`）及 `AvailableToolDefinitions`、`DefaultAllowedTools`。
- **具体工具**：每个工具实现 `types.Tool`，在 `AgentService.registerTools` 中按 config 注册；MCP 工具由 `RegisterMCPTools` 从 MCP 服务动态注册。

### 4.4 Agent Skills（internal/agent/skills）

- **设计**：Progressive Disclosure（渐进式披露）— Level1 元数据进 SystemPrompt，Level2 通过 `read_skill` 按需加载 SKILL.md，Level3 按需读文件或执行脚本。
- **Manager**：初始化时扫描 `skill_dirs`、校验 YAML frontmatter、维护元数据与 Skill 实例；提供 `GetAllMetadata`、`LoadSkill`、`ReadSkillFile`、`ListSkillFiles`、`ExecuteScript`。
- **沙箱**：脚本执行走 `sandbox.Manager`（docker / local / disabled），执行前脚本与参数做安全校验（见 `docs/agent-skills.md`）。

### 4.5 会话与流式（与 Agent 的衔接）

- **SessionService**（如 `session.go`）：在「agent-chat」流程中解析 agent_id，解析/加载 CustomAgent，将 `CustomAgentConfig` 转为运行时 `AgentConfig`（含 SearchTargets、知识库 ID、选中文档等）；为该请求创建**独立 EventBus**；调用 `AgentService.CreateAgentEngine` 得到引擎；`AgentStreamHandler` 订阅该 EventBus；然后调用 `engine.Execute(...)`；Handler 将事件转为 SSE 写入 `StreamManager`。
- **AgentStreamHandler**（`internal/handler/session/agent_stream_handler.go`）：订阅 `EventAgentThought`、`EventAgentToolCall`、`EventAgentToolResult`、`EventAgentReferences`、`EventAgentFinalAnswer`、`EventAgentReflection`、`EventError`、`EventSessionTitle`、`EventAgentComplete`，将事件 append 到 StreamManager，供前端消费。

### 4.6 事件类型（internal/event）

- `EventAgentQuery`、`EventAgentThought`、`EventAgentToolCall`、`EventAgentToolResult`、`EventAgentReflection`、`EventAgentReferences`、`EventAgentFinalAnswer`、`EventAgentComplete` 等，用于驱动流式输出与追踪。

## 5. API 与前端结构

### 5.1 智能体管理 API（docs/api/agent.md）

- `POST /api/v1/agents`：创建自定义智能体。
- `GET /api/v1/agents`：列表（内置 + 自定义）。
- `GET /api/v1/agents/:id`：详情。
- `PUT /api/v1/agents/:id`：更新（内置不可改）。
- `DELETE /api/v1/agents/:id`：删除（仅自定义）。
- `POST /api/v1/agents/:id/copy`：复制（含内置）。
- `GET /api/v1/agents/placeholders`：占位符定义，用于提示词配置。

### 5.2 对话中使用 Agent

- 使用「agent-chat」类接口（见 `docs/api/chat.md`），请求体中传 `agent_id`、`agent_enabled` 等；后端按 agent 配置构建引擎并执行 ReAct/RAG。

### 5.3 前端模块（简要）

- **Agent 管理**：`views/agent/AgentList.vue`、`AgentEditorModal.vue`；`views/settings/AgentSettings.vue`。
- **对话**：`components/AgentSelector.vue`、`AgentAvatar.vue`；`views/chat/components/AgentStreamDisplay.vue` 展示思考、工具调用、最终答案。
- **共享**：`AgentShareSettings.vue`（跨成员/租户共享在应用层与 repository 层实现）。

## 6. 内置智能体与扩展

### 6.1 内置智能体（internal/types/custom_agent.go）

- **builtin-quick-answer**：快速问答，RAG，默认上下文模板与检索/FAQ 策略。
- **builtin-smart-reasoning**：智能推理，ReAct，默认允许工具含 thinking、knowledge_search、grep_chunks 等，可开网络搜索。
- **builtin-data-analyst**：数据分析师，ReAct，仅允许 thinking、todo_write、data_schema、data_analysis，限定 csv/xlsx，启用反思，专用 SystemPrompt（DuckDB/SQL 规范）。

`GetBuiltinAgent(id, tenantID)`、`BuiltinAgentRegistry`、`GetBuiltinAgentIDs()` 用于解析与列表顺序。

### 6.2 扩展方式

- **MCP**：租户配置 MCP 服务，Agent 配置中 `mcp_selection_mode`（all/selected/none）与 `mcp_services`；创建引擎时把选中且启用的 MCP 工具注册到 ToolRegistry。
- **Skills**：仅 smart-reasoning 且配置启用时，加载 Skills 目录、注册 read_skill/execute_skill_script，SystemPrompt 中注入 Skills 元数据。

## 7. 配置与安全要点

- **迭代上限**：`max_iterations` 在 AgentService 中校验（如 ≤ MAX_ITERATIONS），防止死循环。
- **工具白名单**：仅 `allowed_tools` 中的工具被注册并暴露给 LLM。
- **知识库/文档范围**：由 `SearchTargets` 与 `knowledge_bases`/`knowledge_ids` 严格限定检索与数据访问范围。
- **Skills 沙箱**：脚本执行前校验危险命令/模式，执行环境隔离（docker/local/disabled），见 `docs/agent-skills.md`。

## 9. 小结

Agent 整体设计围绕「**双模式（RAG / ReAct）+ 可配置智能体 + 可扩展工具与 Skills**」展开：类型与配置在 `types`，引擎与 ReAct 循环在 `internal/agent`，创建与注册在 `AgentService`，会话与流式在 Session 与 `AgentStreamHandler`，前端通过 Agent 管理页与对话页选择并展示智能体与流式结果。扩展点包括：新内置智能体（注册到 `BuiltinAgentRegistry`）、新工具（实现 `Tool` 并在 `registerTools` 中注册）、MCP 服务、以及 Agent Skills 目录与沙箱配置。
